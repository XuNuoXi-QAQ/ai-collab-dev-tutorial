

## 一、项目背景与核心痛点

### 1.1 背景
在日常运维或AI自动化工作中，AI Agent（如AstrBot）时常需要执行一些需要`sudo`权限的系统级命令（如安装软件、修改配置、重启服务等）。如果让Agent全自动执行，存在巨大的安全风险；如果每次都手动登录服务器执行，效率又太低。

### 1.2 核心痛点
| 痛点 | 描述 |
| :--- | :--- |
| **安全风险** | Agent若拥有完整sudo权限，一旦被滥用或被攻击，后果严重。 |
| **操作繁琐** | 每次需要执行特权操作时，都需要人工登录服务器手动执行。 |
| **审批缺失** | 没有统一的审批流程和操作审计日志，事后难以追溯。 |
| **跨平台需求** | 团队成员可能使用飞书、微信、抖音等不同的办公平台，需要统一入口。 |

### 1.3 项目目标
构建一个**一次开发，多端运行**的小程序，实现：
1. AI Agent提出执行特权命令的请求。
2. 系统自动生成可执行的Shell脚本供人工审核。
3. 用户通过小程序一键审批（同意/拒绝）。
4. 审批通过后，Agent自动执行脚本并回传结果。
5. 所有操作可追溯、可审计。

## 三、核心业务流程

### 3.1 完整流程时序图

```
Agent          AstrBot            Taro小程序          用户
 │                │                   │                │
 │ ① 请求执行特权任务 │                   │                │
 │───────────────>│                   │                │
 │                │ ② 生成Shell脚本     │                │
 │                │ ③ 推送审批通知      │                │
 │                │──────────────────>│                │
 │                │                   │ ④ 展示脚本内容   │
 │                │                   │───────────────>│
 │                │                   │                │ ⑤ 查看并审批
 │                │                   │<───────────────│
 │                │ ⑥ Webhook回调     │                │
 │                │<──────────────────│                │
 │ ⑦ 执行脚本     │                   │                │
 │<───────────────│                   │                │
 │ ⑧ 回传执行结果  │                   │                │
 │───────────────>│                   │                │
 │                │ ⑨ 推送结果通知     │                │
 │                │──────────────────>│                │
 │                │                   │ ⑩ 展示结果      │
 │                │                   │───────────────>│
```

### 3.2 各环节详细说明

| 步骤 | 环节 | 执行方 | 详细说明 |
| :--- | :--- | :--- | :--- |
| ① | 请求执行 | Agent | Agent因业务需求（如需要安装依赖），向AstrBot发起执行特权命令的请求。 |
| ② | 生成脚本 | AstrBot | 根据请求内容，动态生成具体的Shell脚本（如`apt-get install python3`），并暂存。 |
| ③ | 推送通知 | AstrBot | 通过飞书/微信的消息适配器，将脚本内容和审批请求以**消息卡片**形式推送到用户的小程序。 |
| ④ | 展示内容 | Taro小程序 | 在小程序审批页面清晰展示：脚本内容、请求Agent、发起时间、风险提示。 |
| ⑤ | 用户审批 | 用户 | 用户阅读脚本后，点击“同意执行”或“拒绝执行”。 |
| ⑥ | 回调结果 | Taro小程序 | 小程序将审批结果通过AstrBot的Webhook接口回传。 |
| ⑦ | 执行脚本 | Agent | AstrBot收到“同意”指令后，调用Agent执行器，以**普通用户身份+sudo**执行脚本。 |
| ⑧ | 回传结果 | Agent | 脚本执行完毕后，将执行日志（成功/失败/输出）回传给AstrBot。 |
| ⑨ | 推送结果 | AstrBot | 将执行结果推送到小程序，通知用户。 |
| ⑩ | 展示结果 | Taro小程序 | 用户在小程序上查看最终执行结果。 |


## 四、技术选型

| 层级 | 技术方案 | 选型理由 |
| :--- | :--- | :--- |
| **前端框架** | **Taro** | 一套代码编译到飞书/微信/抖音等多平台，社区活跃，支持React/Vue。 |
| **核心后端** | **AstrBot** | 原生支持多平台消息适配器（飞书/微信等），内置Agent执行器，有Webhook能力，开源可扩展。 |
| **内网穿透** | **FRP**（生产） / **ngrok**（测试） | 将内网AstrBot服务暴露到公网，供小程序回调Webhook。 |
| **通信协议** | HTTPS + Token认证 | 保证数据传输加密和接口身份验证。 |
| **权限控制** | sudo + 脚本白名单 | Agent运行用户无完整root权限，仅可通过sudo执行特定脚本。 |
| **审计日志** | AstrBot内置日志 + 数据库 | 记录所有审批请求、用户操作、脚本内容和执行结果。 |


## 五、安全方案设计

安全是本项目的生命线，需要从多个层面进行防护：

### 5.1 身份认证层
| 措施 | 说明 |
| :--- | :--- |
| **Webhook Token** | AstrBot的Webhook接口配置Secret Token，只有携带正确Token的请求才被接受。 |
| **小程序登录鉴权** | 用户需通过各平台（飞书/微信）的OAuth登录，确保操作者身份可识别。 |
| **Agent认证** | Agent与AstrBot之间使用API Key进行身份验证。 |

### 5.2 权限控制层
| 措施 | 说明 |
| :--- | :--- |
| **最小权限原则** | Agent进程以普通用户身份运行，不拥有直接root权限。 |
| **sudo精确控制** | 在`/etc/sudoers`中配置，仅允许Agent用户执行特定命令（如`/usr/bin/apt-get`），而非所有命令。 |
| **脚本白名单** | 后端只允许执行经过模板化生成的脚本，不直接执行任意用户输入的命令。 |

### 5.3 操作审计层
| 措施 | 说明 |
| :--- | :--- |
| **完整日志** | 记录：谁(Who) + 何时(When) + 哪个Agent(Where) + 执行了什么脚本(What) + 结果(Result)。 |
| **脚本快照** | 每次审批的脚本内容完整存储，方便事后审查。 |
| **异常告警** | 对频繁审批、非工作时间操作等异常行为进行告警。 |

### 5.4 网络安全层
| 措施 | 说明 |
| :--- | :--- |
| **HTTPS强制** | 所有对外通信必须使用HTTPS，防止中间人攻击。 |
| **内网隔离** | AstrBot部署在内网，仅通过加密隧道（FRP）暴露必要端口。 |
| **IP白名单** | 如条件允许，限制仅允许小程序平台官方IP段访问Webhook。 |


## 六、部署方案

### 6.1 部署拓扑

```
                    ┌──────────────────┐
                    │   公网云服务器    │
                    │  (FRP服务端)     │
                    └────────┬─────────┘
                             │ 加密隧道
                    ┌────────┴─────────┐
                    │   公司/家庭内网   │
                    │  ┌─────────────┐ │
                    │  │  AstrBot    │ │
                    │  │  + Agent    │ │
                    │  └─────────────┘ │
                    └──────────────────┘
```

### 6.2 部署步骤

| 阶段 | 步骤 | 说明 |
| :--- | :--- | :--- |
| **第一步** | 部署AstrBot | 在内网服务器上安装并配置AstrBot，启用飞书/微信适配器和Webhook功能。 |
| **第二步** | 配置内网穿透 | 使用FRP将AstrBot的Webhook端口（如6185）映射到公网。 |
| **第三步** | 配置HTTPS | 为公网域名配置SSL证书，确保Webhook地址为HTTPS。 |
| **第四步** | 开发Taro小程序 | 使用Taro开发审批页面，配置各平台（飞书/微信/抖音）的编译插件。 |
| **第五步** | 提交审核 | 分别提交飞书、微信、抖音小程序平台审核。 |

## 八、风险与应对

| 风险 | 影响 | 应对措施 |
| :--- | :--- | :--- |
| **小程序平台审核不通过** | 无法上线 | 提前阅读各平台审核规范，确保功能符合要求（尤其是涉及"远程控制"的功能需说明清楚使用场景）。 |
| **内网穿透不稳定** | Webhook回调失败 | 生产环境使用FRP自建服务，配置多节点备份。 |
| **安全漏洞导致权限被滥用** | 严重安全事故 | 严格执行最小权限原则，增加人工审批环节，详细审计日志。 |
| **跨平台兼容性问题** | 某些平台功能异常 | 使用Taro官方插件，针对各平台做充分测试。 |
| **公司网络安全策略限制** | 无法部署内网穿透 | 提前与IT部门沟通，或考虑将AstrBot直接部署在公网云服务器上。 |


## 九、总结

本方案通过 **Taro跨平台小程序** + **AstrBot消息中枢** + **内网穿透** 的组合，构建了一套安全、高效、跨平台的AI Agent权限审批系统。核心思想是：

> **"AI可以建议，但执行必须由人审批"**

这既保留了AI自动化带来的效率提升，又将安全风险的决策权牢牢掌握在人类手中。所有操作可追溯、可审计，真正做到了"让AI帮忙干活，但不让AI替你做主"。
