移动端控制面 + 桌面端能力面”的分布式操作系统雏形。

### 一、整体愿景与核心命题

**目标**：用户打开手机上的一个小程序，就能和电脑上运行的任意“可对话程序”进行交互——包括 OpenCode、豆包、本地大模型、代码助手、自动化脚本等一切可以通过自然语言驱动的工具。

**核心命题**：手机是“遥控器”，电脑是“能力中心”。小程序只负责呈现对话界面和传递指令，真正执行任务的是电脑上运行的各种工具。

**关键挑战**：
- 电脑上的工具各自为政，接口协议不同（有的走 HTTP API，有的是本地进程，有的只提供命令行）
- 手机和电脑之间的通信需要穿透网络
- 用户需要知道“哪个工具能做什么”，系统需要做智能路由


### 二、整体架构：四层模型

这套系统从下到上分为四层，每一层解决一个特定的异构性问题：

**第一层：工具适配层（运行在电脑上）**

这一层是“插件系统”的物理载体。每个电脑上的工具（OpenCode、豆包、本地模型等）都被一个独立的**适配器（Adapter）** 包裹。适配器负责：
- 将工具的原生接口（无论是 HTTP API、WebSocket、本地 RPC 还是命令行）统一转换成标准格式
- 管理工具的生命周期（启动、停止、健康检查）
- 收集工具的能力描述（它能做什么、需要什么参数）

**第二层：电脑端网关（运行在电脑上）**

这是一个轻量级的本地服务，负责：
- 注册和管理所有已安装的适配器
- 接收来自手机的请求，路由到对应的适配器
- 将适配器的响应流式回传给手机
- 处理鉴权和会话管理

**第三层：云信令层（运行在云端）**

这是手机和电脑之间的“信令桥梁”，负责：
- 维护手机与电脑之间的长连接（WebSocket）
- 转发对话请求和响应流
- 管理设备绑定关系（哪个手机绑定了哪台电脑）
- 可选：做轻量级的意图识别，帮助用户选择工具

**第四层：小程序前端层（运行在手机上）**

这是用户直接打交道的界面，用 Taro 开发，覆盖微信、支付宝、抖音、飞书四个平台。它负责：
- 展示对话界面
- 将用户输入发送到云信令层
- 实时展示电脑端返回的流式响应
- 展示可用工具列表，帮助用户选择或让系统自动路由


### 三、技术栈选型

**小程序前端（Taro）** ：
- 使用 **Taro** 框架，一套代码编译到微信、支付宝、抖音、飞书四个平台
- 桌面端如需配套应用，Taro 可通过 Electron 或 Tauri 方式覆盖
- 状态管理选用 Zustand 或 Jotai，处理多会话、流式消息等复杂状态

**电脑端网关与适配器（Rust）** ：
- 沿用你熟悉的 **Rust + Actix-web** 技术栈
- 每个适配器是一个独立的 Rust 模块或动态库，实现统一的 `ToolAdapter` Trait
- 网关本身是一个轻量级 HTTP/WebSocket 服务，监听本地端口

**云信令层（Rust）** ：
- 使用 **Actix-web + WebSocket** 支持，处理手机与电脑的长连接
- 使用 **Redis** 做会话状态和设备绑定信息的短暂存储
- 可选使用 **PostgreSQL** 做用户、设备、会话历史的持久化

**通信协议**：
- 手机 ↔ 云信令层：**WebSocket**（实时双向通信）
- 云信令层 ↔ 电脑网关：**WebSocket** 或 **长轮询**（取决于网络环境）
- 电脑网关 ↔ 各工具适配器：**进程内调用**（适配器作为库）或 **HTTP**（适配器作为独立服务）

**核心依赖**：
- `reqwest` + `reqwest_eventsource`：处理与外部工具的 HTTP 请求和 SSE 流
- `tokio_tungstenite`：WebSocket 服务端和客户端
- `serde` + `serde_json`：所有跨层数据序列化
- `tracing`：全链路日志追踪


### 四、通信协议设计：统一的消息格式

为了让手机、云信令层、电脑网关、各个工具之间能顺畅对话，需要定义一套**统一的消息协议**。这套协议借鉴了 OpenAI 的 Chat Completion 格式和 AG-UI 协议的设计思想。

**统一请求格式**：
```
{
  "message_id": "唯一ID",
  "session_id": "会话ID", 
  "tool": "opencode" | "doubao" | "local_llama" | "auto",  // 指定工具或自动路由
  "messages": [  // 对话历史，标准格式
    {"role": "user", "content": "帮我写一个排序算法"},
    {"role": "assistant", "content": "好的，以下是..."}
  ],
  "stream": true,  // 是否流式返回
  "parameters": {}  // 工具特定参数（温度、模型等）
}
```

**统一响应格式（流式）** ：
```
// 每个 chunk 是一个 JSON 对象
{"type": "text", "content": "这是", "done": false}
{"type": "text", "content": "一段流式", "done": false}
{"type": "text", "content": "回复", "done": true}
{"type": "tool_call", "name": "execute_code", "args": {...}}  // 工具调用事件
{"type": "status", "state": "thinking"}  // 状态事件
```

这套格式的优势在于：
- **与 OpenAI API 兼容**：OpenCode 本身提供 OpenAI 兼容的 API，可以直接映射
- **支持流式**：通过 Server-Sent Events 或 WebSocket 实现实时响应
- **支持工具调用**：可以传递“执行代码”、“打开文件”等指令
- **状态可见**：用户能看到“思考中”、“执行中”、“完成”等状态


### 五、插件（适配器）设计准则

每个电脑上的工具对应一个适配器。适配器的设计遵循以下准则：

**准则一：每个适配器独立成模块**

OpenCode 一个适配器，豆包一个适配器，本地 Ollama 一个适配器……每个适配器放在独立的 Rust 模块或独立的代码仓库中。增加一个新工具，就是新增一个适配器，**不修改任何已有代码**。

**准则二：适配器只做“翻译”**

适配器的唯一职责是**协议转换**：
- 将统一的请求格式转换成工具原生 API 的请求格式
- 将工具原生 API 的响应转换成统一的响应格式

适配器**不做**业务判断（如“这个请求该不该执行”），**不做**用户鉴权（由上层做），**不做**会话管理（由上层做）。

**准则三：适配器声明自己的能力**

每个适配器在注册时，要声明自己“能做什么”：
- 工具名称和描述
- 支持的参数（模型选择、温度等）
- 能力标签（如 `can_execute_code`、`can_read_files`、`can_browse_web`）

这样上层可以根据用户意图自动选择合适的工具，而不需要用户手动指定。

**准则四：适配器支持流式和非流式两种模式**

有些工具（如 OpenCode）原生支持流式输出，有些可能不支持。适配器要能处理这两种情况，对不支持的做“伪流式”（攒够一批再发）。


### 六、智能路由：如何帮用户找到对的工具

当用户说“帮我写个快排算法”时，系统怎么知道该用 OpenCode 还是豆包？

**方案：三层路由决策**

1. **用户显式指定**：用户在小程序界面上直接选择“使用 OpenCode”或“使用豆包”
2. **系统自动推荐**：根据用户输入的关键词和适配器声明的能力标签做匹配。例如“写代码”匹配 `can_execute_code` 标签
3. **兜底策略**：如果都不匹配或用户没指定，使用默认工具（如 OpenCode）

路由决策可以在**云信令层**做，也可以在**电脑网关层**做。建议放在电脑网关层，因为电脑网关知道本地安装了哪些工具，云信令层只需要转发请求即可。


### 七、网络穿透与设备绑定

这是整套系统最关键的工程难题——**手机怎么找到电脑**？

**方案：云端信令 + NAT 穿透**

1. **设备注册**：电脑端网关启动后，向云信令层注册自己，建立 WebSocket 长连接。云信令层给这台电脑分配一个唯一设备 ID，并生成一个绑定码（如 6 位数字）。
2. **手机绑定**：用户在小程序上输入绑定码，云信令层将手机会话与电脑设备关联。
3. **消息转发**：手机发送消息 → 云信令层 → 通过 WebSocket 推送给电脑网关 → 电脑网关调用适配器 → 响应原路返回。

**备选方案**：如果电脑有公网 IP 或做了端口映射，也可以让手机直接连接电脑的 IP + 端口，但 WebSocket + 云端信令的方案更通用，适用于绝大多数家庭网络环境。


### 八、安全与鉴权

**三层安全机制**：

1. **设备绑定鉴权**：绑定码是一次性的、有时效的（如 5 分钟有效），防止未授权设备绑定
2. **会话鉴权**：每个手机会话有自己的 Token，云信令层验证每个请求的合法性
3. **工具级鉴权**：某些敏感工具（如能执行代码、能操作文件系统）可以要求二次确认，或在电脑端弹窗让用户手动批准

### 十、实施路线图

**第一阶段（MVP）** ：
- 实现电脑端网关 + 一个适配器（如 OpenCode）
- 实现云信令层的基础 WebSocket 转发
- 实现 Taro 小程序的基础对话界面
- 完成设备绑定流程
- 目标：用户能在手机上通过小程序与电脑上的 OpenCode 对话

**第二阶段（扩展）** ：
- 增加豆包、Ollama 等更多适配器
- 实现智能路由（自动匹配工具）
- 增加会话历史持久化
- 支持流式响应

**第三阶段（完善）** ：
- 增加工具调用能力（让 AI 能执行代码、操作文件）
- 增加多电脑管理（绑定多台设备，切换使用）
- 增加桌面端配套应用（用 Taro + Electron 实现电脑上的控制面板）
- 完善安全机制和错误处理


### 总结

这套方案的核心理念是 **“手机做交互，电脑做计算，云端做桥梁”** 。通过四层架构（工具适配层 → 电脑网关 → 云信令层 → 小程序前端）和统一的通信协议，将电脑上各种异构的 AI 工具编织成一个可被手机小程序统一调用的能力网络。

你之前设计的“多平台认证 + 多厂商 Agent 适配”架构，正好可以无缝对接到这套体系的**工具适配层**和**云信令层**——原本适配的是各家云厂商的 API，现在适配的是电脑上运行的各种本地工具。本质上是同一套“适配器模式”在不同粒度上的应用。

你想先从哪个环节开始深入？是电脑端网关的设计，还是云信令层的网络穿透方案，或者是 Taro 前端的多端适配？
