## ——从“指令式绘图”到“意图式生成”的技术跨越

**报告人**：何静（北京航空航天大学副教授）

**报告日期**：2026年6月


## 摘要

本报告系统梳理了基于OpenAI Codex构建的Codex for CAD智能绘图助手在工程设计与CAD制图领域的应用评测。报告首先分析了传统工程设计工作流中计算、绘图、改图三个阶段的效率瓶颈与数据断层问题；其次阐述了Codex for CAD的技术原理与核心能力——基于CAD领域专用数据（AutoLISP、VBA、.NET API、脚本宏、图元数据库等）微调而成的AI引擎，能够将自然语言描述和工程逻辑直接转化为可执行的CAD绘图命令。报告详细介绍了四种接入CAD的方式（DWG直出、脚本生成、MCP调用、Skill封装）及其适用场景，提出了Skill的三级成熟度模型，并以基坑支护工程为测试载体，完整演示了从“计算书生成”到“三维模型构建”再到“多目标复合修改”的全流程实操。评测表明，Codex for CAD在数据整理与表格输出、标准构件图初稿生成、文字标注与批量图层整理等高度重复性任务中表现优异，但其能力边界亦需明确——结构安全的最终决策、施工图标准合规性判断等仍需注册工程师把关。报告最后论证了该方法的跨行业可迁移性，并对未来演进方向进行了展望。


## 第一章 引言

### 1.1 研究背景

工程设计行业正经历一场由人工智能驱动的深刻变革。传统的CAD（计算机辅助设计）工作流程长期依赖工程师手动操作——鼠标点选、键盘命令输入、坐标定义——本质上是一种“指令式绘图”范式。工程师在计算软件（如理正基坑、PKPM等）与CAD绘图软件之间反复切换，手动将计算结果誊抄到图纸上，不仅效率低下，而且极易引入人为误差。

与此同时，大语言模型（Large Language Model, LLM）技术的快速发展为这一困局提供了新的破局思路。OpenAI Codex作为GPT系列模型的代码生成变体，经过大量源代码和自然语言的训练，具备将自然语言描述转化为可执行代码的能力。当Codex与CAD领域数据（AutoLISP、VBA、.NET API、脚本宏、图元数据库等）进一步微调结合后，便形成了具备工程语义理解能力的“Codex for CAD”AI引擎。

### 1.2 工程痛点分析

在传统工程设计工作流中，工程师面临三大核心痛点：

**计算阶段的痛点**：计算软件（如理正基坑）输出大量数据，工程师必须手动将关键参数誊抄到CAD图纸上。一旦某一数据抄错，整张图纸就必须返工重做。这种“人肉搬运”不仅耗时，更是设计错误的主要来源之一。

**绘图阶段的痛点**：以中型基坑工程为例，往往需要绘制十几张甚至数十张剖面图，每张图的绘制步骤高度重复——布置支护桩、绘制土层分界线、标注尺寸、添加配筋信息——工程师被迫陷入大量重复性劳动。

**改图阶段的痛点**：设计变更是工程实践的常态。业主或设计方微调一个参数（如桩径从1.0m改为1.2m），所有关联的尺寸标注、配筋信息、材料统计都必须手动逐一更新，牵一发而动全身。

### 1.3 研究目标与报告结构

本报告旨在系统评估Codex for CAD在工程设计与CAD制图领域的应用价值与能力边界。研究以基坑支护工程为测试载体，围绕四种Codex接入CAD的方式（DWG直出、脚本生成、MCP调用、Skill封装）分别测试其适用场景、输出形式与核心能力。报告结构如下：第二章介绍Codex for CAD的技术原理与核心能力；第三章详述四种接入方式的对比评测；第四章提出Skill的三级成熟度模型；第五章以基坑支护工程为例，完整演示计算、绘图、改图三个阶段的实操过程；第六章总结能力边界与跨行业可迁移性；第七章进行总结与展望。


## 第二章 Codex for CAD：技术原理与核心能力

### 2.1 什么是Codex for CAD

Codex for CAD是基于OpenAI Codex，经过CAD领域专用数据（包括AutoLISP、VBA、.NET API、脚本宏、图元数据库等）微调而成的AI引擎。其核心功能是将自然语言描述和工程逻辑直接转化为可执行的CAD绘图命令和参数化图元。

与传统CAD的“指令式绘图”不同，Codex for CAD实现了向“意图式生成”的技术跨越——工程师不再需要记忆繁琐的命令和坐标，只需用自然语言描述设计意图（如“绘制一排桩，桩径800毫米，间距1.6米”），AI便能理解语义并自动生成对应的CAD图形。

### 2.2 三大核心能力

**（一）工程语义解析**

Codex for CAD能够理解“桩径800mm”、“间距1.60米”、“混凝土强度C30”等专业工程术语。这种语义理解能力并非简单的关键词匹配，而是建立在对工程领域知识深度理解基础上的概念识别与关系抽取。例如，当工程师提到“排桩+锚索支护”时，AI不仅能识别“排桩”和“锚索”两个独立构件，还能理解它们之间的空间组合关系。

**（二）逻辑编码**

AI能够自动生成绘制直线、圆弧、填充、标注、图块插入等底层绘图代码。这意味着工程师只需描述“画什么”，而不必关心“怎么画”——具体的坐标计算、图元生成、图层分配等底层逻辑全部由AI自动完成。

**（三）上下文学习与风格延续**

Codex for CAD能够延续既有图纸的绘图风格，包括图层规范、标注样式、线型比例等。这对于需要保持企业制图标准一致性的设计院来说尤为重要。

### 2.3 与传统方式的对比

| 对比维度 | 传统交互式CAD | 简单宏/脚本 | Codex + CAD |
|---|---|---|---|
| 输入方式 | 鼠标点选、键盘命令 | 预录顺序操作 | 自然语言+工程参数 |
| 灵活性 | 高，但依赖人工操作 | 低，参数变化需重写 | 高，根据描述动态生成代码 |
| 对计算结果的响应 | 人工读取、手动输入 | 无法自动读取 | 可直接读取计算文本或表格数据 |
| 工程师负担 | 双手不离鼠标键盘，重复劳动 | 需要编程知识 | 聚焦设计意图，描述即可 |
| 输出产物 | .dwg图纸 | 固定套路 | 完整的、可编辑的.dwg图纸 |

Codex for CAD并非简单的“语音绘图”工具，而是一个具备工程语义理解能力的“AI绘图助手”，其核心价值在于让CAD从“工具”进化为“能听懂工程师计算结果的智能伙伴”。

### 2.4 技术基础：从Codex到工程AI

OpenAI Codex本质上是将自然语言（如英语）转化为可运行代码的AI模型。在CAD领域的应用中，Codex需要处理的不仅是通用编程语言（如Python），还包括AutoCAD专用的脚本语言（如AutoLISP、VBA）以及DWG文件格式的底层操作。通过在这些CAD领域数据上的进一步微调，Codex for CAD获得了对工程制图规范和绘图逻辑的深度理解。

值得关注的是，类似的AI辅助CAD自动化正在成为行业趋势。研究表明，新一代生成式AI基础模型预计将自动化设计师约90%的CAD几何创建工作量。MCP（Model Context Protocol，模型上下文协议）作为一种开放标准，正在为AI模型连接数据源和工具提供标准化框架。Autodesk也已宣布将开发支持MCP的服务，进一步推动AI与CAD的深度融合。


## 第三章 四种接入方式的系统评测

Codex for CAD并非只有一种固定的使用模式。根据不同的应用场景、技术条件和目标需求，存在四种由浅入深、各具特色的接入方式。本次评测围绕四种方式分别测试其适用场景、输出形式和核心能力，以DWG方式看结果，脚本方式看过程，MCP方式看交互，Skill方式看复用。

### 3.1 方式一：Codex直接生成DWG文件

**基本思路**：让Codex根据自然语言需求，直接生成CAD可打开的图纸文件（如DWG文件）。典型流程为：自然语言描述 → Codex生成绘图代码 → 代码生成DWG → CAD打开检查。

**评测定位**：用于测试Codex是否能从工程参数直接生成CAD成果，主要输出形式为.dwg图纸文件，重点验证参数识别、几何生成和基础图元表达能力。

**优点**：
- 上手简单，无需复杂的环境配置
- 不一定需要直接连接AutoCAD
- 适合快速验证Codex绘图能力

**局限**：
- 复杂DWG对象支持有限
- 图层、标注样式、企业制图标准需要额外控制
- 修改已有DWG文件时能力相对受限

**评测侧重点**：更关注“能不能直接生成可打开的CAD图纸”，以及“图形是否完整、比例是否正确、标注是否可用”。

### 3.2 方式二：Codex生成CAD脚本或插件代码

**基本思路**：让Codex生成AutoLISP、VBA、Python或.NET插件代码，工程师在CAD中运行脚本完成绘图或改图。典型流程为：自然语言需求 → Codex生成AutoLISP/Python/.NET代码 → CAD加载运行 → 自动绘图/改图。

**评测定位**：用于测试Codex是否能生成可复现、可调试的绘图逻辑，主要输出形式为AutoCAD脚本，重点验证图层控制、标注生成、尺寸表达和脚本复用能力。

**优点**：
- 更接近真实CAD工作流
- 方便保存为企业内部工具
- 可做成参数化绘图模板

**局限**：
- 需要一定CAD二次开发环境
- 代码需要调试
- 不同CAD版本兼容性较差

**评测侧重点**：更关注“能不能把绘图逻辑写成可运行脚本”，以及“脚本是否稳定、图元是否规范、后续是否可复用”。

值得注意的是，用不同的AI工具生成AutoLISP脚本的效果存在差异。实测表明，使用Claude Code生成的AutoLISP脚本注释更为完整，后续团队维护和修改的可读性更好——这一点对于需要长期维护的企业级应用尤为重要。

### 3.3 方式三：通过MCP连接CAD工具

**基本思路**：MCP（Model Context Protocol）是一种让AI应用连接外部工具和数据的开放标准。Codex通过MCP协议直接与AutoCAD建立连接，实现AI对CAD软件的实时操控。典型流程为：Codex → MCP → Server → CAD命令/图纸文件/构件库 → 执行绘图或改图。

**评测定位**：用于测试Codex是否能从AutoCAD环境中读取、定位、修改已有图纸，主要形式为直接操作DWG并另存新文件，重点验证图元定位、局部修改、联动调整和改图闭环能力。

**优点**：
- 自动化程度高
- 可以连接多个工具
- 未来可扩展为工程Agent
- 适合企业级集成

**局限**：
- 配置复杂度较高
- 需要开发或适配CAD MCP Server
- 权限、安全、文件操作需要严格控制

**评测侧重点**：更关注“能不能真正操作已有CAD文件”，以及“是否能准确选中对象、避免误改漏改”。

目前，已有开发者实现了基于MCP的AutoCAD集成服务器，允许通过Claude等大语言模型与AutoCAD进行自然语言交互。AutoCAD LT的AutoLISP MCP Server也已出现，可通过自然语言控制AutoCAD LT 2024/2025。这些实践表明，MCP正在成为AI与CAD深度融合的关键技术桥梁。

### 3.4 方式四：Skill封装流程

**基本思路**：将CAD绘图流程、提示词模板、企业图层标准、常用脚本、构件库等封装成可复用的Skill，让Codex在处理类似任务时自动遵循固定流程。

**评测定位**：用于测试Codex是否能借助Skill完成标准化、参数化CAD任务，主要输出为Skill生成的CAD几何成果/模型/图纸，重点验证标准流程复用、参数化建模和企业能力沉淀。

**优点**：
- 复用性强
- 便于沉淀设计经验
- 可以减少每次重新写提示词的成本
- 更适合标准化、流程化任务

**局限**：
- 对非标准化任务适应性较弱
- Skill质量取决于内部规范、脚本和样例积累

**评测侧重点**：更关注“能不能把某类任务变成固定能力”，以及“是否适合批量复用、是否能沉淀为设计院工具链”。

### 3.5 四种方式的综合对比

| 实现方式 | 核心逻辑 | 技术门槛 | 自动化程度 | 最适合场景 |
|---|---|---|---|---|
| DWG直出 | AI直接输出成品图纸文件 | 极低 | 高（一次性） | 个人验证/快速原型 |
| 脚本生成 | AI生成中间代码，人工加载运行 | 中等 | 中 | 日常提效/内部工具 |
| MCP调用 | AI直接接管CAD软件 | 较高 | 极高 | 企业级集成/工程Agent |
| Skill封装 | 规范与模板打包为固定工作流 | 中等 | 高 | 标准化制图/能力沉淀 |


## 第四章 Skill的三级成熟度模型

在四种接入方式中，Skill封装是最具企业级应用价值的一种。根据封装深度和能力成熟度，Skill可分为三个递进层级。

### 4.1 第一级：无代码Skill

**定义**：最基础的Skill形式，通过输入提示词模板、图层规范、命名规则和标注要求，让Codex在每次执行任务时遵循固定规则，无需重复解释。

**典型组成**：
- 提示词模板（标准化任务描述框架）
- 企业图层标准（如“桩-图层”用特定颜色和线型）
- 命名规则（如图块命名规范）
- 标注要求（文字高度、字体样式等）

**适用场景**：任务逻辑相对简单、主要依赖自然语言描述即可完成的工作，如标准化的文字标注、图层整理等。

### 4.2 第二级：低代码Skill

**定义**：在无代码Skill的基础上，引入已验证的AutoLISP脚本和Excel模板，Codex可根据需要调用这些预置资源。

**典型组成**：
- 无代码Skill的全部内容
- 验证通过的AutoLISP脚本库
- Excel参数模板（如材料表、坐标表）

**适用场景**：需要执行较为复杂的几何运算或批量处理的任务，如参数化构件生成、批量编号等。

### 4.3 第三级：企业专属Skill

**定义**：将设计院的所有内部规范、标准、构件库和通用绘图流程完整打包，形成针对特定任务的定制化能力，如“基坑支护CAD Skill”或“配筋图生成Skill”。

**典型组成**：
- 低代码Skill的全部内容
- 企业级构件库（标准化图块、节点大样）
- 完整的制图流程规范
- 质量检查标准

**适用场景**：企业级标准化生产，要求输出成果完全符合内部质量体系的任务。

### 4.4 Skill封装的工程价值

Skill封装的核心价值在于将“一次性提示”转化为“可复用的企业资产”。传统的AI使用方式中，每次任务都需要工程师重新编写提示词，不仅效率低下，而且质量参差不齐。通过Skill封装，设计经验、制图规范和脚本资源得以沉淀为组织的长期能力，新人上手即可调用成熟的Skill完成标准化任务，大幅降低培训成本和质量波动。


## 第五章 三个阶段实操评测

本次评测以基坑支护工程为测试载体，完整覆盖了从“计算”到“绘图”再到“改图”的工程全流程。以下分别详述三个阶段的实操过程、提示词设计、输出结果与评测发现。

### 5.1 计算阶段：从参数提取到完整计算书

#### 5.1.1 任务目标与评测定位

计算阶段的任务是让Codex基于上传的工程勘察文件或计算书，提取土体参数，完成土压力系数计算，并最终生成一份结构完整的基坑支护计算书。

评测定位：测试Codex是否能从工程参数直接完成结构化计算任务。这一阶段的核心能力验证包括参数识别、公式应用、计算过程可追溯性和文档规范化输出。

#### 5.1.2 提示词设计（土压力系数计算）

**提示词模板**：

> 请基于我上传的文件/桌面上XX文件，提取【土体类型】和【土体参数】，计算【目标土压力系数】。要求：
> 1. 明确采用的计算理论或公式，如【Rankine理论/Coulomb理论/经验公式】。
> 2. 展示完整计算过程，不能只给最终结果。
> 3. 输出表格，包含[层号]、[土类名称]、[内摩擦角φ(°)]、[黏聚力c(kPa)]、[Ka]、[Kp]，保留三位小数。
> 4. 将计算结果以表格形式输出到Word文档，保存到【位置】，文件命名为“【XXXX计算结果.docx】”。

**提示词示例**：

> 请基于我桌面上《1剖面计算书》，从中提取“土层参数”表格。计算每一层土的：主动土压力系数 Ka、被动土压力系数 Kp。展示完整计算过程，不能只给最终结果。输出表格，包含层号、土类名称、内摩擦角φ(°)、黏聚力c(kPa)、Ka、Kp，保留三位小数，将计算结果以表格形式输出到Word文档，保存到我的桌面，文件命名为“【土压力系数计算结果.docx】”。

#### 5.1.3 输出示例

依据标准：《建筑基坑支护技术规程》JGJ-120-2012。计算方法：朗肯土压力理论。

Ka = tan²(45° - φ/2)，Kp = tan²(45° + φ/2)

| 层号 | 土类名称 | 内摩擦角φ(°) | 黏聚力c(kPa) | Ka | Kp |
|---|---|---|---|---|---|
| 1 | 杂填土 | 16.000 | 29.100 | 0.568 | 1.761 |
| 2 | 细砂 | 25.000 | 0.000 | 0.406 | 2.464 |
| 3 | 圆砾 | 35.000 | 0.000 | 0.271 | 3.690 |
| ... | ... | ... | ... | ... | ... |

注：当c>0时，主动土压力强度需叠加-2c√Ka，表中Ka仅用于计算土压力系数部分。

#### 5.1.4 评测发现

**表现评价**：计算阶段是三个阶段中表现最稳定、最成熟的。原因在于，计算本质上是结构化数据处理——参数作为输入、数字和表格作为输出，规则明确、结果可验，这正是AI最擅长的任务类型。

**关键洞察**：提示词设计中有一个极为重要的原则——“防幻觉”约束。AI的默认行为是“填满所有空白”，如果不在提示词中明确划定边界，它会自行编造看似合理的数据。因此，必须在提示词中明确要求：“对资料中没有提供或需要人工复核的内容，用【需人工复核】标注，不得自行编造。”这一原则贯穿了所有阶段的提示词设计。

#### 5.1.5 计算书生成（进阶任务）

在完成单点参数计算后，进一步测试了Codex生成完整计算书的能力：

**提示词模板**：

> 请基于我上传的文件/桌面上XX文件，生成一份【基坑支护完整计算书】。要求：
> 1. 提取【工程基本信息】、【支护结构参数】、【土层参数】、【地下水位】、【地面超载】和【支锚信息】。
> 2. 按照计算书格式，依次完成【工程概况】、【计算条件】、【土压力计算】、【支锚计算】、【桩身内力与配筋复核】、【稳定性验算】和【结论】。
> 3. 每一部分都要保留必要的计算过程和结果表，不要只输出结论。
> 4. 对资料中没有提供或需要人工复核的内容，用【需人工复核】标注，不得自行编造。
> 5. 将完整计算书输出为Word文档，保存到【位置】，文件命名为【完整计算书生成结果.docx】。

**评测发现**：Codex能够基于已有计算书和工程参数，快速重组生成结构完整的计算书初稿，涵盖工程概况、计算条件、土压力计算、支锚计算、桩身内力与配筋复核、稳定性验算（整体稳定、抗隆起、突涌稳定、嵌固深度构造验算）和最终结论等全部章节。输出格式规范、计算过程可追溯，特别适合用于计算成果整理、复核汇报和文档标准化。

### 5.2 绘图阶段：从构件到体系

绘图阶段是AI价值体现最为直观的环节，但同时也是能力边界最为明显的环节。评测覆盖了两个递进层次：单一构件（桩配筋示意图）和完整体系（三维支护模型）。

#### 5.2.1 任务一：桩配筋示意图（2D构件）

**采用方式**：text-to-cad Skill封装方式

**提示词模板**：

> 请调用【text-to-cad Skill】，基于我上传的文件/桌面上XX文件，生成【桩配筋示意图】。要求：
> 1. 提取【桩径】、【保护层厚度】、【纵筋规格】、【箍筋规格】和【加强箍筋规格】。
> 2. 将配筋参数转化为可编辑CAD图形。
> 3. 绘制【桩截面外轮廓】、【保护层线】、【纵筋】、【箍筋】和【钢筋说明】。
> 4. 对无法自动还原的施工图表达细节，用【待人工确认】标注。
> 5. 输出.dwg文件，并生成一份图纸生成说明。

**提示词示例**：

> 请调用【text-to-cad Skill】，基于我桌面上的【剖面计算书】，生成桩配筋示意图。已知桩径1.000m，保护层厚度50mm，纵筋为HRB400 18E25，箍筋为HPB300 d8@200，加强箍筋为HRB400 E18@2000。请生成可编辑CAD图形，包含桩截面外轮廓、保护层线、18根主筋、箍筋和钢筋文字说明。对于无法完全符合施工图表达规范的内容，请标注为【待人工确认】，输出.dwg文件，并生成一份图纸生成说明，放在桌面。

**输入→输出映射**：

| 输入参数 | 输出图元 |
|---|---|
| 桩径 1.000m | 桩截面外轮廓（圆形） |
| 保护层厚度 50mm | 保护层线（同心圆） |
| 纵筋 HRB400 18E25 | 18根主筋沿圆周均匀布置 |
| 箍筋 HPB300 d8@200 | 箍筋示意 |
| 加强箍筋 HRB400 E18@2000 | 加强箍示意 |

**评测发现**：
- Codex/Skill正确识别了18根主筋并进行了圆周均布计算
- 输出示例显示了完整的配筋信息：纵筋HRB400 18E25、箍筋HPB300 d8@200、加强箍筋HRB400 E18@2000
- 延续了“防幻觉”设计原则，对无法完全符合施工图表达规范的内容标注【待人工确认】
- text-to-cad Skill支持将多种语言的参数转化为可编辑CAD图形，但施工图标注规范和构造细节仍需人工校核

**值得注意的细节**：提示词示例中的“18E25”在工程制图中通常表示直径25mm的HRB400钢筋共18根。但“E”在钢筋符号体系中通常代表抗震钢筋（HRB400E），而非普通HRB400——这提示在实际使用中需要明确钢筋牌号的完整表述，避免歧义。

#### 5.2.2 任务二：基坑支护体系三维示意模型（3D体系）

**采用方式**：text-to-cad Skill封装方式

**提示词模板**：

> 请调用【text-to-cad Skill】，生成【基坑支护体系三维示意模型】。要求：
> 1. 根据输入的【支护形式】、【基坑尺寸】和【构件参数】，生成三维示意模型。
> 2. 模型中应包含【基坑开挖面】、【支护桩/地下连续墙】、【冠梁】、【锚索/内支撑】等主要构件。
> 3. 各构件应保持基本空间关系正确，能够体现支护体系的整体构成。
> 4. 模型要求达到正式施工图精度，要保证结构清晰、层次分明。
> 5. 输出.dwg文件，并生成一份图纸生成说明。

**提示词示例**：

> 请调用【text-to-cad Skill】，生成一个基坑支护体系三维示意模型。支护形式为排桩+锚索支护，基坑深度17.200m，支护桩直径1.000m，桩间距1.400m，冠梁尺寸为1.100m×0.700m，共设置4道锚索，锚索总长22.00m，入射角15°。请在模型中体现基坑开挖面、排桩、冠梁和4道锚索的空间关系，形成一个完整的三维支护体系示意模型。模型要求达到正式施工图精度，要保证结构清晰、层次分明，输出.dwg文件，并生成一份图纸生成说明。

**构件参数汇总**：

| 构件 | 参数 |
|---|---|
| 基坑开挖面 | 深度17.200m |
| 支护桩 | 直径1.000m，间距1.400m |
| 冠梁 | 1.100m × 0.700m |
| 锚索 | 4道，总长22.00m，入射角15° |

**评测发现**：
- 从“单一构件”到“完整体系”的维度跃升成功实现——模型包含基坑开挖面、排桩、冠梁和4道锚索等全部主要构件
- 各构件保持了正确的空间相对关系（桩沿基坑边线布置、锚索从桩间斜向打入、冠梁连接桩顶）
- 对精度的要求从“示意”提升到“正式施工图精度”
- 输出物包括DWG文件和图纸生成说明

**关键洞察：2D与3D的精度差异**：

评测揭示了一个重要规律：三维示意模型和二维施工图对AI生成的要求截然不同。
- **三维示意模型**：空间关系大致正确即可接受，适合向客户或非专业人士展示支护方案的整体构成
- **二维施工图**：每条线的坐标必须绝对精确，容差极低——目前AI尚不能直接生成可供施工使用的二维图纸，标注规范和构造细节仍需人工校核

这一发现明确了当前AI能力的合理定位：三维模型可用于方案展示和概念设计阶段，而二维施工图的最终出图仍需注册工程师把关。

### 5.3 改图阶段：多目标复合修改

改图阶段是MCP方式价值体现最为充分的场景。它要求AI不仅能“画”，还要能“在既有图纸上精准地改”——这是从“生成工具”到“设计助手”的关键跨越。

#### 5.3.1 任务目标

基于既有CAD图纸（基坑支护平面布置图），根据设计变更要求完成一组多目标复合修改——新增排水沟示意线、护栏示意线、排水方向箭头、文字说明和引线。

**采用方式**：MCP调用AutoCAD

#### 5.3.2 提示词设计

**提示词模板**：

> 通过MCP连接AutoCAD，打开我提供的【CAD文件路径/图纸名称】，根据【设计说明/变更要求】完成一组多目标复合修改。
>
> 新增/修改内容：
> 1. 在【位置】新增【对象一】。
> 2. 在【位置】新增【对象二】。
> 3. 增加【箭头/符号/标注/文字说明】。
> 4. 必要时新建或使用【图层名称】图层。
>
> 绘图要求：
> 1. 新增对象应与原图比例、线型、文字高度和标注风格协调。
> 2. 不遮挡原有支护结构线、尺寸线、监测点和图框内容。
> 3. 图层归属清晰，文字放在TEXT图层，辅助符号放在AUXILIARY图层，结构或排水线放在【指定图层】。
> 4. 修改完成后保存为【输出文件名.dwg】。
> 5. 输出新增对象清单和复合修改说明。

**提示词示例**：

> 请通过MCP连接AutoCAD，打开20240424基坑支护平面布置图.dwg，根据设计说明中“基坑顶部设置排水沟，护栏和排水沟之间设置一定坡度”的要求，完成一组多目标复合修改。
>
> 新增内容：
> 1. 在基坑平面图北侧支护边线外侧新增一条顶部排水沟示意图线。
> 2. 在排水沟外侧新增一条护栏示意图线。
> 3. 在排水沟线上增加排水方向箭头。
> 4. 增加文字说明：“基坑顶部排水沟，坡向集水方向”。
> 5. 必要时增加一条引线，将文字说明指向排水沟。
>
> 绘图要求：
> 6. 排水沟线和护栏线应与原基坑边线平行，距离保持清晰可读。
> 7. 排水沟线、护栏线、箭头和文字说明宜放在STRUCTURE图层或新建DRAINAGE图层。
> 8. 新增内容不得遮挡原有支护结构线、尺寸线、监测点和图框内容。
> 9. 修改完成后保存在桌面，文件命名为03_多目标复合修改_排水沟护栏.dwg。
> 10. 输出新增对象清单，包括图层、对象类型、作用说明，以及无法确定处的【待人工确认】标注。

#### 5.3.3 任务复杂度分析

本次改图任务包含5项关联的新增/修改操作：
1. 新增排水沟示意线（需与基坑边线平行、保持适当距离）
2. 新增护栏示意线（在排水沟外侧）
3. 增加排水方向箭头（在排水沟线上）
4. 增加文字说明
5. 增加引线（将文字指向排水沟）

这些操作不是孤立的，而是相互关联的复合修改——每项新增内容的位置、图层归属和空间关系都需要统筹考虑。

#### 5.3.4 评测发现

**表现评价**：改图阶段呈现出最清晰的“能力边界”。
- 简单任务（单图元精确定位编辑）→ 表现稳定
- 局部联动编辑（关联尺寸同步更新）→ 基本可行
- 需工程逻辑理解的任务 → 出错率明显上升

**关键注意事项**：
> 若图纸缺少明确的比例尺，AI会从图面自行量测再偏移——实际精度须人工复核。这是MCP改图场景中最容易被忽略的安全隐患。

**MCP改图的独特价值**：
- 能够打开既有图纸，理解原有图面布局
- 能够在**不破坏原有内容**的前提下精准插入新对象
- 新增对象能与原图风格协调（比例、线型、文字高度、标注风格）
- 输出新增对象清单（含图层、对象类型、作用说明），实现可追溯的修改记录

**“语义定位”的复杂性**：
提示词中的“在基坑平面图北侧支护边线外侧”涉及两个层面的定位：
- **语义定位**：“北侧”=哪个区域？“外侧”=哪一侧？
- **几何定位**：边线的偏移距离是多少？

这比“在坐标(100,200)处画一个圆”要复杂得多，因为它要求MCP先识别图纸中的方向基准和目标图元，再计算具体坐标——这是“语义理解”与“几何操作”的深度融合。

#### 5.3.5 与Skill方式的对比

| 对比维度 | 绘图阶段（Skill） | 改图阶段（MCP） |
|---|---|---|
| 输入 | 工程参数 | 既有图纸+变更需求 |
| 核心能力 | 从0生成几何 | 在既有图面上精准插入新对象 |
| 关键约束 | 参数→图元的映射准确性 | 空间避让、风格协调、不遮挡 |
| 输出物 | DWG+生成说明 | DWG+新增对象清单+修改说明 |


## 第六章 能力边界与行业迁移

### 6.1 当前能力的优势区间

基于三个阶段的系统评测，Codex for CAD在当前阶段表现最为优异的任务类型可归纳为：

**（一）数据整理与表格输出**
- 从勘察报告、计算书中提取参数
- 结构化数据的计算与汇总
- 规范化的表格生成与输出

**（二）标准构件图初稿生成**
- 桩配筋示意图
- 标准剖面图
- 构件大样图

**（三）文字标注与批量图层整理**
- 图面标准化处理
- 批量编号与标注
- 图层归属整理

**（四）绘图前的参数一致性检查**
- 计算书与图纸参数的对照验证
- 数据完整性与合理性初筛

这些任务的共性特征包括：**高度重复性、规则明确、输入输出结构清晰**——交给AI能显著节省工程师的时间，且出错风险可控。

### 6.2 当前能力的边界

与此同时，以下类型的任务在现阶段仍需人工判断和干预：

**（一）代码适用性判断**
- 不同CAD版本的兼容性差异
- 脚本在不同环境中的运行稳定性

**（二）结构安全的最终决策**
- 配筋方案的合理性判断
- 稳定性验算结果的工程评判
- **不可模糊的底线**：AI可辅助整理和复核，但不能取代注册工程师的设计责任

**（三）施工图标准合规性**
- 企业制图标准的细节判断
- 施工图表达规范的完整性

**（四）跨图纸联动修改**
- 涉及多张图纸间关联关系的全局一致性
- 复杂设计变更的连锁影响分析

### 6.3 跨行业可迁移性

本次评测以基坑支护工程为测试载体，但Codex for CAD的底层逻辑具有广泛的跨行业可迁移性。凡涉及参数化绘图、计算结果转图纸、批量图纸编辑的场景，均可参考本报告所建立的方法论框架。

以下为各设计行业的可迁移任务与推荐方式对照：

| 行业 | 可迁移任务 | 适合采用方式 | 典型应用 |
|---|---|---|---|
| 建筑设计 | 平面布置图、门窗表、节点详图、面积指标复核 | DWG/脚本/MCP | 快速生成方案图、批量修改房间名称和尺寸标注 |
| 结构设计 | 梁柱配筋图、节点详图、构件编号、钢筋表整理 | 脚本/MCP/Skill | 标准构件出图、配筋示意图生成、图表一致性检查 |
| 岩土工程 | 支护剖面、土层图、锚杆锚索布置、稳定性验算图 | DWG/脚本/MCP | 计算书参数转图纸、改图联动、支护方案表达 |
| 市政道路 | 道路横断面、管线综合、检查井布置、标高标注 | 脚本/MCP | 批量生成断面图、检查管线冲突和标高一致性 |
| 给排水设计 | 管线平面图、系统图、井表、管径标注 | MCP/Skill | 批量替换管径、检查管线编号和材料表一致性 |

| 电气设计 | 配电箱系统图、桥架布置、回路编号、设备表 | 脚本/MCP/Skill | 批量编号、回路表检查、图纸与清单联动 |
| 水利工程 | 河道断面、护坡结构、闸门构件、边坡示意图 | DWG/脚本/Skill | 标准断面图生成、构件示意图和参数化绘图 |

| 室内设计 | 平面布局、家具布置、节点详图、材料标注 | DWG/MCP | 快速生成布局方案、批量调整标注和图层 |

**关键洞察**：虽然各行业业务领域不同，但底层任务类型是相通的——平面布置图（建筑）≈管线平面图（给排水）≈风管布置（暖通）→ 都属于“二维平面系统图”；构件编号（结构）≈井表/管径标注（给排水）≈回路编号（电气）→ 都属于“图面与表格一致性管理”。这种“任务类型跨行业共通”的特性，使得Codex for CAD的方法论可以成为工程设计领域的通用AI基础设施。


## 第七章 总结与展望

### 7.1 核心结论

通过本次系统评测，可以得出以下核心结论：

**第一，Codex for CAD实现了从“指令式绘图”到“意图式生成”的技术跨越**。工程师不再需要记忆繁琐的命令和坐标，只需用自然语言描述设计意图，AI便能理解语义并自动生成对应的CAD图形。这一转变的本质，是将工程师从“工具操作者”解放为“设计意图定义者”。

**第二，四种接入方式各有所长，应根据任务类型灵活选择**。DWG直出适合快速验证，脚本生成适合日常提效，MCP调用适合企业级集成，Skill封装适合标准化生产。不存在“最好”的方式，只有“最合适”的方式。

**第三，Skill封装是企业级应用的最优路径**。通过将设计经验、制图规范和脚本资源沉淀为可复用的Skill，设计院可以将“个人能力”转化为“组织资产”，实现标准化、批量化的高质量出图。

**第四，AI的能力边界必须明确**。Codex for CAD在数据整理、标准构件生成、批量标注等高度重复性任务中表现优异，但结构安全的最终决策、施工图标准的合规性判断等仍需注册工程师把关。AI是“助手”，不是“替代者”。

### 7.2 长期价值

Codex for CAD的长期价值不在于一次性出图，而在于将**计算→绘图→复核→修改**串联成**可追溯、可复用的工作流**。它打通了从“计算文档”到“CAD图纸”的数据管道，从源头切断了人工誊抄带来的误差链，让工程师能够将更多精力投入到创造性设计工作中。

### 7.3 未来演进方向

基于本次评测的发现，Codex for CAD的未来演进可朝以下方向推进：

**方向一：从“生成文件”到“直接操作CAD”** 。当前MCP方式已初步实现AI对CAD的实时操控。未来随着MCP生态的成熟和AutoCAD官方MCP服务的推出，AI将能更深度地融入CAD工作流，实现真正的“对话式设计”。

**方向二：从“单点绘图”到“全流程自动化”** 。未来的工程智能体将能够完成“解析计算书→提取参数→CAD输出→改图→变更说明”的完整闭环，实现工程设计全流程的智能化。

**方向三：从“被动执行”到“主动检查”** 。AI将不仅执行绘图指令，还能主动检查图纸的一致性、参数的完整性、图面与计算书的吻合度，充当设计师的“智能质检员”。

**方向四：从“通用工具”到“企业专属Skill”** 。随着各设计院将内部规范、标准和经验沉淀为专属Skill，Codex for CAD将从通用AI工具演进为符合各企业质量体系的定制化生产力平台。


## 附录：提示词设计原则速查

基于本次评测的实践经验，总结以下提示词设计核心原则：

**原则一：明确指定计算依据和理论**。如“依据《建筑基坑支护技术规程》JGJ-120-2012，采用朗肯土压力理论”——防止AI自行选择不合适的计算方法。

**原则二：要求展示完整计算过程**。“不能只给最终结果”——确保计算可追溯、可复核。

**原则三：强制“防幻觉”约束**。“对资料中没有提供或需要人工复核的内容，用【需人工复核】标注，不得自行编造”——划定AI的“创作边界”。

**原则四：明确输出格式与保存位置**。“输出表格，包含[层号]、[土类名称]……保存到桌面，文件命名为‘XXX.docx’”——确保输出成果可直接使用。

**原则五：分层定义精度要求**。区分“示意模型”（空间关系正确即可）和“施工图精度”（坐标必须精确）的不同标准，避免预期偏差。

**原则六：要求输出辅助文档**。“输出一份图纸生成说明”或“输出新增对象清单”——确保AI的工作可审计、可追溯。
