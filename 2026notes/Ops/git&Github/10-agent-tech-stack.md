# Agent 技术栈详解

> 深入了解 AI Agent 的核心技术组件，从大语言模型到工具调用，从记忆系统到规划推理。

---

## 🧠 大语言模型（LLM）：Agent 的大脑

### LLM 在 Agent 中的作用

大语言模型是 Agent 的核心，相当于人类的大脑。它负责：

| 能力 | 说明 |
|------|------|
| **理解能力** | 理解用户的自然语言指令 |
| **推理能力** | 分析问题，推导解决方案 |
| **规划能力** | 制定行动计划和步骤 |
| **生成能力** | 生成代码、文本、回答等 |
| **决策能力** | 在多个选项中做出选择 |

### 主流 LLM 对比

| 模型 | 特点 | 适合的 Agent 场景 |
|------|------|------------------|
| **GPT-4o** | 能力全面，多模态，工具调用强 | 通用型 Agent，复杂任务 |
| **Claude 3.5** | 长文本处理强，代码能力好 | 文档处理，代码 Agent |
| **Gemini 1.5** | 多模态强，长上下文 | 多媒体处理，研究型 Agent |
| **DeepSeek** | 国产，代码能力强 | 国内场景，开发 Agent |
| **Ollama 本地模型** | 可离线，隐私安全 | 敏感数据，离线场景 |

### 选择 LLM 的考虑因素

1. **能力水平**：是否能处理你的任务复杂度
2. **上下文长度**：需要处理多少文本
3. **工具调用支持**：是否支持函数调用
4. **响应速度**：对延迟的要求
5. **成本预算**：API 调用成本
6. **隐私安全**：数据是否可以上传

## 🔧 工具调用（Tool Use）：Agent 的双手

### 什么是工具调用

工具调用让 Agent 能够使用外部工具来完成任务，而不仅仅是靠模型自身的知识。

**类比**：
- 没有工具的 Agent ≈ 只会空想的人
- 有工具的 Agent ≈ 会使用工具的实干家

### 常见的工具类型

#### 1. 代码执行工具

```python
# 让 Agent 能够运行代码
def execute_code(code: str) -> str:
    """执行 Python 代码并返回结果"""
    # ... 执行逻辑
    return output
```

**用途**：
- 数据计算和分析
- 算法验证
- 快速原型验证

#### 2. 文件操作工具

```python
def read_file(path: str) -> str:
    """读取文件内容"""
    
def write_file(path: str, content: str) -> None:
    """写入文件内容"""
    
def list_directory(path: str) -> list:
    """列出目录内容"""
```

**用途**：
- 代码开发
- 文档编写
- 项目管理

#### 3. 网络搜索工具

```python
def web_search(query: str) -> list:
    """搜索网络信息"""
    
def fetch_webpage(url: str) -> str:
    """获取网页内容"""
```

**用途**：
- 信息检索
- 最新资讯获取
- 研究调研

#### 4. API 调用工具

```python
def call_api(url: str, method: str, params: dict) -> dict:
    """调用外部 API"""
```

**用途**：
- 数据库查询
- 第三方服务调用
- 系统操作

#### 5. 版本控制工具

```python
def git_commit(message: str) -> None:
    """提交代码"""
    
def git_push() -> None:
    """推送到远程仓库"""
```

**用途**：
- 代码版本管理
- 协作开发

### 工具调用的工作原理

#### Function Calling（函数调用）

这是目前最主流的工具调用方式：

```
1. 用户提问
   ↓
2. LLM 判断是否需要调用工具
   ↓
3. 需要工具 → 生成工具调用参数
   ↓
4. 执行工具，获取结果
   ↓
5. 将工具结果返回给 LLM
   ↓
6. LLM 根据结果生成最终回答
```

#### 示例：用工具查询天气

```
用户：北京今天天气怎么样？

LLM 思考：我需要调用天气查询工具
→ 调用 get_weather(city="北京")

工具返回：{ "temp": "28°C", "weather": "晴", "humidity": "45%" }

LLM 生成回答：北京今天天气晴朗，气温 28°C，湿度 45%，适合出门。
```

## 📝 记忆系统：Agent 的长期记忆

### 为什么需要记忆系统

LLM 的上下文窗口是有限的，无法记住所有历史信息。记忆系统让 Agent 能够：

1. **记住历史对话**：长期记忆用户偏好
2. **积累知识**：从经验中学习
3. **保持一致性**：前后回答保持一致
4. **个性化服务**：根据用户习惯调整

### 记忆的层次

| 记忆类型 | 作用 | 存储方式 | 类比 |
|----------|------|----------|------|
| **短期记忆** | 当前会话上下文 | LLM 上下文窗口 | 工作记忆 |
| **中期记忆** | 最近的对话历史 | 对话数据库 | 近期记忆 |
| **长期记忆** | 知识和经验 | 向量数据库 | 知识库 |

### 向量数据库（Vector Database）

长期记忆通常使用向量数据库存储：

**工作原理**：
1. 将文本转换为向量（Embedding）
2. 存储到向量数据库
3. 查询时，将问题也转为向量
4. 搜索最相似的记忆

**常见的向量数据库**：
- Chroma（轻量级，适合本地）
- Pinecone（云端服务）
- Weaviate（开源，功能丰富）
- Milvus（高性能，企业级）

### 记忆的检索策略

#### 1. 相似度检索

根据语义相似度检索相关记忆：

```
问题：如何用 Git 撤销提交？
    ↓
检索相关记忆：
  - Git 基础命令
  - 版本回退方法
  - reset 和 revert 的区别
    ↓
将相关记忆加入上下文
```

#### 2. 时间加权检索

最近的记忆权重更高：

```
今天的对话 > 昨天的对话 > 一周前的对话
```

#### 3. 重要性加权

重要的记忆权重更高：

```
用户明确说"记住这个" > 普通对话 > 闲聊内容
```

## 🎯 规划与推理能力：Agent 的思考

### 为什么需要规划

复杂任务不能一步完成，需要 Agent 能够：

1. **分解任务**：将大任务拆成小步骤
2. **制定计划**：确定执行顺序
3. **动态调整**：根据情况变化调整计划
4. **反思优化**：从错误中学习

### 常见的规划方法

#### 1. Chain of Thought（思维链）

让模型一步步思考：

```
用户：15 + 27 × 3 等于多少？

思维链：
1. 先算乘法：27 × 3 = 81
2. 再算加法：15 + 81 = 96
3. 所以答案是 96
```

#### 2. Tree of Thoughts（思维树）

探索多种可能性，选择最优路径：

```
问题：如何提高代码质量？

方案 A：增加测试
  ├─ 单元测试
  ├─ 集成测试
  └─ 端到端测试

方案 B：代码审查
  ├─ 人工审查
  ├─ 自动化工具
  └─ AI 审查

方案 C：重构优化
  └─ ...

→ 综合评估，选择最佳方案组合
```

#### 3. ReAct（推理 + 行动）

边思考边行动，根据结果调整：

```
Thought: 用户想知道今天的天气，我需要先知道用户所在城市
Action: 询问用户所在城市
Observation: 用户说在北京

Thought: 现在知道城市了，我可以查询天气
Action: 调用天气 API 查询北京天气
Observation: 返回天气数据

Thought: 有了天气数据，可以回答用户了
Final Answer: 北京今天天气...
```

### 规划的挑战与应对

| 挑战 | 说明 | 应对策略 |
|------|------|----------|
| **计划过于死板** | 遇到意外就卡住 | 动态调整，保留灵活性 |
| **规划太耗时** | 想太多不行动 | 先粗后细，边做边调 |
| **容易走偏** | 偏离原始目标 | 定期检查，回归目标 |
| **错误累积** | 一步错步步错 | 检查点，及时纠偏 |

## 👁️ 多模态感知能力：Agent 的感官

### 什么是多模态

多模态意味着 Agent 不仅能处理文本，还能处理：

- 🖼️ **图像**：识别图片内容
- 🎵 **音频**：理解语音和音乐
- 🎬 **视频**：分析视频内容
- 📊 **图表**：理解数据可视化

### 多模态的应用场景

#### 1. 代码开发

- 截图中的错误识别
- UI 设计图转代码
- 图表数据分析

#### 2. 文档处理

- 图片中的文字识别（OCR）
- 手写笔记数字化
- PDF 文档理解

#### 3. 创意设计

- 参考图风格分析
- 设计稿评审
- 视觉效果评估

### 主流多模态模型

| 模型 | 模态 | 特点 |
|------|------|------|
| **GPT-4o** | 文本、图像、音频 | 能力全面，响应快 |
| **Claude 3 Opus** | 文本、图像 | 长文本，高智能 |
| **Gemini 1.5 Pro** | 文本、图像、音频、视频 | 长上下文，多模态强 |
| **Qwen-VL** | 文本、图像 | 国产，开源 |

## 🏗️ Agent 的完整技术架构

### 经典 Agent 架构

```
┌─────────────────────────────────────────┐
│              用户交互层                   │
│        （接收输入，展示输出）              │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│              规划层                      │
│    （任务分解、计划制定、动态调整）        │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│              记忆层                      │
│  （短期记忆、长期记忆、向量检索）          │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│              工具层                      │
│  （代码执行、文件操作、网络搜索、API）     │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│            大语言模型（LLM）              │
│         （理解、推理、生成、决策）         │
└─────────────────────────────────────────┘
```

### 各层之间的协作

```
1. 用户输入 → 用户交互层
   ↓
2. 规划层分析任务，制定计划
   ↓
3. 从记忆层检索相关信息
   ↓
4. 选择合适的工具执行
   ↓
5. LLM 处理工具返回的结果
   ↓
6. 检查是否完成目标
   ├─ 未完成 → 回到第 2 步，继续规划
   └─ 已完成 → 返回结果给用户
```

## 🚀 技术选型建议

### 入门级（快速上手）

- **LLM**：GPT-3.5-turbo / Claude 3 Haiku
- **工具**：基础的函数调用
- **记忆**：简单的对话历史
- **适用**：简单的个人助手、原型验证

### 进阶级（生产可用）

- **LLM**：GPT-4o / Claude 3.5 Sonnet
- **工具**：丰富的工具集，代码执行
- **记忆**：向量数据库，长期记忆
- **规划**：ReAct 模式，基础规划
- **适用**：专业助手、开发工具

### 专家级（复杂系统）

- **LLM**：多个模型组合，MoA 架构
- **工具**：定制化工具，插件系统
- **记忆**：多层记忆，知识图谱
- **规划**：复杂规划，多 Agent 协作
- **适用**：企业级系统，复杂任务
