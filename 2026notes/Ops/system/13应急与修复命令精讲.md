# 应急与修复命令精讲

应急与修复命令是系统管理员的“最后一道防线”。当系统出现严重问题时，这组命令帮助你快速恢复系统、诊断故障、清理资源。

---

## 一、核心概念

### 系统状态层级
| 状态 | 描述 | 适用场景 |
|------|------|----------|
| **正常运行** | 系统正常提供服务 | 日常使用 |
| **救援模式** | 单用户模式，最小化服务 | 修复启动问题 |
| **紧急模式** | 最小化环境，仅挂载根分区 | 严重问题修复 |
| **单用户模式** | root 用户，无网络 | 重置密码等 |

### 应急操作的优先级
1. **诊断问题** → `journalctl`、`dmesg`
2. **尝试修复** → `systemctl`、`pacman`
3. **清理资源** → `drop_caches`、`paccache`
4. **重启恢复** → `reboot`、`shutdown`

## 2. `reboot` / `shutdown` / `systemctl` — 重启与关机

### 命令格式
```bash
# 重启
sudo reboot                          # 立即重启
sudo systemctl reboot                # 通过 systemd 重启
sudo shutdown -r now                 # 立即重启
sudo shutdown -r +5                  # 5分钟后重启
sudo shutdown -r 23:00               # 23:00 重启

# 关机
sudo shutdown -h now                 # 立即关机
sudo poweroff                        # 关机
sudo systemctl poweroff              # 通过 systemd 关机
sudo shutdown -h +10                 # 10分钟后关机
```

### 参数说明
| 参数 | 含义 |
|------|------|
| `-h` | 关机（halt） |
| `-r` | 重启（reboot） |
| `-c` | 取消已计划的关机/重启 |
| `+N` | N 分钟后执行 |
| `now` | 立即执行 |

### 实际应用
```bash
# 立即重启
sudo reboot

# 立即关机
sudo shutdown -h now

# 5分钟后重启
sudo shutdown -r +5

# 取消计划的重启
sudo shutdown -c

# 向所有用户发送消息后关机
sudo shutdown -h +10 "系统将在10分钟后关闭，请保存工作"
```

## 3. `sync && echo 3 > /proc/sys/vm/drop_caches` — 清理系统缓存

### 命令格式
```bash
sudo sync && sudo echo 3 > /proc/sys/vm/drop_caches
```

### 参数说明
| 值 | 含义 |
|----|------|
| `1` | 清理页面缓存（page cache） |
| `2` | 清理目录项和 inode 缓存 |
| `3` | 清理所有缓存（1 + 2） |

### 工作流程
```
sync          → 将内存中的脏数据写入磁盘
echo 3 > /proc/sys/vm/drop_caches → 清理缓存
```

### 实际应用
```bash
# 清理所有缓存（最常用）
sudo sync && sudo echo 3 > /proc/sys/vm/drop_caches

# 只清理页面缓存
sudo sync && sudo echo 1 > /proc/sys/vm/drop_caches

# 查看缓存使用情况
free -h

# 清理前后对比
free -h && sudo sync && sudo echo 3 > /proc/sys/vm/drop_caches && free -h
```

## 4. `sudo pacman -Syu --noconfirm` — 无确认更新系统

### 命令格式
```bash
sudo pacman -Syu --noconfirm       # 无确认更新
sudo pacman -Syu --noconfirm --ignore gdm  # 忽略特定包
```

### 参数说明
| 参数 | 含义 |
|------|------|
| `--noconfirm` | 自动确认所有提示 |
| `--ignore 包名` | 跳过指定包的更新 |
| `--overwrite '*'` | 允许覆盖文件（危险！） |

### 实际应用
```bash
# 自动更新（脚本中使用）
sudo pacman -Syu --noconfirm

# 更新时忽略 gdm（防止更新后无法登录）
sudo pacman -Syu --noconfirm --ignore gdm

# 只更新特定包
sudo pacman -Syu --noconfirm docker htop
```

## 5. `systemctl rescue` / `systemctl emergency` — 救援与紧急模式

### 命令格式
```bash
sudo systemctl rescue        # 进入救援模式
sudo systemctl emergency     # 进入紧急模式
```

### 模式对比
| 模式 | 服务 | 网络 | 挂载 | 适用场景 |
|------|------|------|------|----------|
| **救援模式** | 最小化服务 | ❌ | 所有分区 | 修复启动问题、重置密码 |
| **紧急模式** | 无服务 | ❌ | 仅根分区（只读） | 严重损坏、修复文件系统 |

### 实际应用
```bash
# 进入救援模式
sudo systemctl rescue

# 进入紧急模式
sudo systemctl emergency

# 从救援模式退出（恢复正常启动）
exit
```

## 6. `journalctl -p err -b -1` — 查看上次启动的错误

### 命令格式
```bash
journalctl -p err -b -1         # 查看上次启动的错误
journalctl -p err -b 0          # 查看本次启动的错误
journalctl -p err -b -1 | tail -20  # 最近20条
```

### 参数说明
| 参数 | 含义 |
|------|------|
| `-p err` | 只显示错误级别及以上的日志 |
| `-b -1` | 上一次启动 |
| `-b 0` | 当前启动 |
| `-b -2` | 前两次启动 |

### 实际应用
```bash
# 查看上次启动的所有错误
journalctl -p err -b -1

# 查看本次启动的错误
journalctl -p err -b 0

# 查看上次启动的最近20条错误
journalctl -p err -b -1 | tail -20

# 查看上次启动的严重错误
journalctl -p crit -b -1
```

## 7. `dmesg -T | grep -i error` — 查看内核错误

### 命令格式
```bash
dmesg -T | grep -i error           # 查看内核错误
dmesg -T | grep -i "fail"          # 查看失败信息
dmesg -T | tail -20                # 最近20条
```

### 实际应用
```bash
# 查看内核错误
dmesg -T | grep -i error

# 查看硬件错误
dmesg -T | grep -i "hardware"

# 查看 USB 错误
dmesg -T | grep -i usb | grep -i error

# 查看磁盘错误
dmesg -T | grep -i "sda" | grep -i error
```

## 8. `lsof +L1` — 查找已删除但仍打开的文件

### 命令格式
```bash
sudo lsof +L1              # 查找所有已删除但未释放的文件
sudo lsof +L1 | grep deleted  # 过滤
```

### 什么是已删除但仍打开的文件？
当一个文件被 `rm` 删除，但仍有进程在使用它时，文件实际上没有被释放，磁盘空间也不会被回收。`lsof +L1` 可以找出这些文件。

### 输出示例
```
COMMAND   PID   USER   FD   TYPE DEVICE SIZE/OFF NLINK  NODE NAME
python   1234  nuoxi    3u   REG   8,2  1048576     0 12345 /tmp/tmpfile (deleted)
```

### 实际应用
```bash
# 查找所有已删除未释放的文件
sudo lsof +L1

# 查看特定进程
sudo lsof +L1 -p 1234

# 释放文件（重启进程或清空文件）
# 方法1：重启进程
sudo systemctl restart 服务名

# 方法2：清空文件（慎用）
sudo truncate -s 0 /proc/1234/fd/3
```

## 9. `df -i` — 查看 inode 使用情况

### 命令格式
```bash
df -i              # 查看 inode 使用情况
df -i /            # 查看根分区 inode
df -iT             # 显示文件系统类型
```

### 什么是 inode？
inode 是文件系统中存储文件元数据的结构。每个文件占用一个 inode。当 inode 用尽时，即使磁盘空间还有剩余，也无法创建新文件。

### 输出示例
```bash
df -i
```
```
文件系统         Inodes  IUsed   IFree IUse% 挂载点
/dev/sda2       3932160 123456 3808704    3% /
```

### 关键指标
| 指标 | 含义 |
|------|------|
| `Inodes` | 总 inode 数量 |
| `IUsed` | 已使用 inode 数量 |
| `IFree` | 剩余 inode 数量 |
| `IUse%` | inode 使用率（>80% 需关注） |

### 实际应用
```bash
# 检查 inode 使用情况
df -i

# 查找占用大量 inode 的目录
find / -type d -exec ls -l {} \; 2>/dev/null | wc -l

# 清理小文件释放 inode
find /tmp -type f -mtime +7 -delete
```

## 10. `mount -o remount,ro /` — 重新挂载根分区为只读

### 命令格式
```bash
sudo mount -o remount,ro /        # 根分区只读
sudo mount -o remount,rw /        # 根分区读写
sudo mount -o remount,ro /home    # /home 只读
```

### 使用场景
- 进行系统维护前保护数据
- 紧急模式下修复文件系统
- 防止意外写入

### 实际应用
```bash
# 将根分区挂载为只读
sudo mount -o remount,ro /

# 恢复读写
sudo mount -o remount,rw /

# 检查挂载状态
mount | grep " / "
```

## 💡 实战场景

### 场景1：系统卡死，需要清理缓存
```bash
# 1. 查看内存使用
free -h

# 2. 清理缓存
sudo sync && sudo echo 3 > /proc/sys/vm/drop_caches

# 3. 再次查看
free -h
```

### 场景2：查看上次启动的故障
```bash
# 查看上次启动的所有错误
journalctl -p err -b -1

# 查看上次启动的最近50条错误
journalctl -p err -b -1 | tail -50
```

### 场景3：磁盘空间不足
```bash
# 1. 查看磁盘和 inode
df -h && df -i

# 2. 查找大文件
du -h --max-depth=1 / | sort -rh | head -10

# 3. 清理缓存
sudo paccache -r -k 3

# 4. 清理日志
sudo journalctl --vacuum-size=500M

# 5. 查找已删除未释放的文件
sudo lsof +L1
```

### 场景4：无法登录，需要进入救援模式
```bash
# 1. 进入救援模式
sudo systemctl rescue

# 2. 修复问题
# 检查并修复文件系统
fsck /dev/sda2

# 重置密码
passwd

# 3. 退出恢复正常启动
exit
```

## ⚠️ 安全提醒

1. **`rm -rf /` 绝对禁止**：永远不要执行 `sudo rm -rf /`。
2. **`drop_caches` 会暂时影响性能**：清理缓存后，系统可能需要重新加载数据。
3. **救援模式需要 root 权限**：确保你记得 root 密码。
4. **`lsof +L1` 找到的文件不要随意删除**：确认进程不再需要后再处理。
5. **在 `/etc/fstab` 中不要随意添加挂载选项**：可能影响系统启动。
6. **测试应急命令前备份数据**：建议先在虚拟机或测试环境中练习。
