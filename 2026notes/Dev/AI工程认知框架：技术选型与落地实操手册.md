## 1. 为什么需要一套新的分类体系

### 1.1 当前困境

你在日常工作中是否遇到过以下情况：

- **情况A**：老板说“我们上个AI”，你打开搜索引擎，发现有大模型、视觉模型、语音模型、3D模型……几十个方向，完全不知道从哪里入手。
- **情况B**：你看了一篇热门的AI论文，里面提到“Diffusion Model”，你花了两周搞懂了原理，结果发现手头的项目只需要用一个现成的API就能解决。
- **情况C**：你被问到“你是做AI哪一块的？”，你试图回答，但发现“视觉”、“自然语言”、“生成”这些标签根本不能准确描述你在做的事情。
- **情况D**：你同时接触了LLM、NeRF、YOLO、XGBoost、Sora、Agent，你觉得它们之间完全没有共同语言，每种都需要从头学起。

这些问题的根源在于：**当前AI领域的分类方式是非正交的、重叠的、混乱的。**

### 1.2 根本原因

市面上常见的分类方式存在以下问题：

| 分类方式 | 问题 | 举例 |
| :--- | :--- | :--- |
| 按功能分（生成/识别） | 同一模型可能兼具两种功能，边界模糊 | GPT既能续写也能分类，SD能生成也能修复 |
| 按架构分（CNN/Transformer） | 同一任务可用不同架构实现，架构不决定任务本质 | 图像识别可用CNN也可用ViT |
| 按厂商/生态分 | 技术本质被商业包装掩盖 | “OpenAI系”包含完全不同的技术路线 |
| 按应用场景分（自动驾驶/医疗） | 同一技术可跨场景复用，场景分类无助于技术理解 | 目标检测既用于自动驾驶也用于安防 |
| 按训练方式分（监督/自监督/强化学习） | 同一任务可用多种训练方式，不决定任务本质 | GPT用了自监督+监督+RLHF混合训练 |

**结果**：你学到的每一个新概念都是孤立的，无法被整合到一个统一的框架里。每出现一个新名词，你都得从头理解一次。你的知识是“点状”的，不是“网状”的。

### 1.3 解决方案：双轴正交分类体系

本报告建立两个**完全正交**的分类坐标轴：

| 坐标轴 | 解决的问题 | 维度 | 核心逻辑 |
| :--- | :--- | :--- | :--- |
| **原子能力轴** | “这个AI在干什么？” | 5种 | 以AI回答的根本问题为锚点，与数据无关 |
| **数据模态轴** | “这个AI在处理什么？” | 9类 | 以数据的本质结构为锚点，与任务无关 |

**为什么正交很重要**：因为正交意味着不重叠、不依赖。任意一个AI系统，一旦确定了它“在干什么”和“在处理什么”，它的技术本质就被唯一确定了——不会出现“它既是A又是B”的混淆。

**类比理解**：
- 这就像用“功能类型”（轿车/SUV/卡车）和“动力类型”（燃油/电动/混动）两个轴来分类汽车。如果只用一个轴，就会混淆；用两个轴，每辆车都有唯一坐标。
- 本框架中，“原子能力”相当于功能类型，“数据模态”相当于动力类型。两者完全独立，组合后唯一确定技术方案。

> **核心理念**：当你把“干什么”和“对什么干”分开，AI的混乱分类就彻底清晰了。


## 2. 第一坐标轴：五原子能力——它在回答什么根本问题

无论数据是文本、图像、音频、点云还是4D时空，AI所做的全部事情，可以归结为回答**五个根本问题**。这五个问题是正交且完备的——互不重叠，且从“感知外部世界”到“改变外部世界”构成了完整的智能闭环。

| 原子能力 | 根本问题 | 工程本质 | 对应名词 |
| :--- | :--- | :--- | :--- |
| **感知** | “现在是什么？” | 从数据里提取标签/坐标/数值 | 图像分类、目标检测、语音识别、YOLO、ResNet |
| **理解** | “中间缺了什么？” | 从碎片恢复完整结构 | 3D重建、图像修复、NeRF、4DGS、BERT |
| **预测** | “接下来是什么？” | 根据历史外推未来 | GPT续写、股价预测、Sora、LSTM |
| **推演** | “如果……会怎样？” | 内部模拟、搜索最优路径 | AlphaGo、o1、物理引擎、Agent决策 |
| **执行** | “我去改变它” | 把决策变成真实操作 | 工具调用、ROS、机械臂指令 |

这五个能力串联起来，就是著名的**OODA闭环**（观察-定向-决策-行动），由美国空军上校约翰·博伊德提出，后被广泛用于认知系统、军事战略和AI领域。任何智能体的智能活动，都可以被分解到这个闭环中。

### 2.1 感知（Perception）——“现在是什么？”

**定义**：从原始数据中提取结构化信息。这是整个智能闭环的“眼睛和耳朵”，也是AI最基础、最成熟的能力。

**输入输出示例**：

| 输入 | 输出 | 任务类型 | 现实案例 |
| :--- | :--- | :--- | :--- |
| 一张图片 | “猫” | 图像分类 | 手机相册自动分类 |
| 一张图片 | “猫在坐标(100,200)” | 目标检测 | 自动驾驶识别行人 |
| 一段音频 | “你好”文本 | 语音识别（ASR） | 智能音箱听写 |
| 一个点云 | “这是一把椅子” | 点云分类 | 机器人抓取前识别物体 |
| 一段文本 | 情感标签“正面” | 情感分析 | 舆情监控系统 |
| 一排传感器读数 | 温度值“37.5℃” | 回归预测 | 工业异常检测 |

**工程本质**：特征提取 + 映射。将高维输入数据“压扁”为低维语义向量，然后映射到目标空间。整个过程可以表示为：`原始数据 → 特征向量 → 输出标签/数值`。

**典型技术路线**：
- 图像：CNN（卷积神经网络）、ViT（视觉Transformer）
- 点云：PointNet、PointNet++、3D卷积
- 音频：时序特征提取（MFCC）、CNN、Transformer
- 文本：Transformer编码器（BERT类）

**代表模型/工具**：

| 子任务 | 代表模型 | 推理速度 |
| :--- | :--- | :--- |
| 图像分类 | ResNet、EfficientNet、ViT | 毫秒级 |
| 目标检测 | YOLO系列、Faster R-CNN、DETR | 毫秒级 |
| 语音识别 | Whisper、SenseVoice | 实时 |
| 点云处理 | PointNet++、PointMLP | 毫秒级 |
| 文本分类 | BERT、RoBERTa | 毫秒级 |

**工程注意点**：
- 感知层的核心代码通常只有几行：`model.predict(data)`
- 工程痛点往往不在模型结构，而在**数据预处理**和**标签质量**。模型的精度天花板由数据质量决定，而非模型大小。
- 当模型精度不达标时，80%的根因是训练数据与推理数据的分布不一致（数据漂移），而非模型不够大。
- 硬件瓶颈：算力（FLOPS）。

**典型代码骨架**：
```python
# 图像分类示例
import torch
from torchvision import models, transforms

transform = transforms.Compose([
    transforms.Resize((224, 224)),
    transforms.ToTensor(),
    transforms.Normalize(mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225])
])

model = models.resnet50(pretrained=True)
model.eval()

with torch.no_grad():
    output = model(image_tensor)
    predicted_class = output.argmax(dim=1)
```

### 2.2 理解（Understanding）——“中间缺了什么？”

**定义**：从碎片化、带噪声、不完整的数据中，恢复出完整的底层表征。这是构建“内心世界模型”的核心能力——它让AI不仅能“看见”，还能“脑补”。

**输入输出示例**：

| 输入 | 输出 | 任务类型 | 现实案例 |
| :--- | :--- | :--- | :--- |
| 缺词的句子 | 填补的词 | 掩码语言建模 | 搜索引擎补全查询 |
| 被遮挡的图像 | 完整图像 | 图像修复 | 老照片修复 |
| 稀疏视角照片（10张） | 完整3D模型 | 3D重建 | 商品三维展示 |
| 环绕拍摄的视频 | 动态3D场景 | 4D重建 | 体育赛事三维回放 |

**与感知的区分**：

| | 感知 | 理解 |
| :--- | :--- | :--- |
| 输入 | 完整、清晰的数据 | 不完整、带噪声、碎片化数据 |
| 任务 | 提取标签/数值 | 恢复完整结构 |
| 比喻 | 直接读表：“现在是3:15” | 表盘碎了，看齿轮推断：“这是计时器，指向3:15” |

**关键技术路线**：

| 路线 | 核心思想 | 代表方法 |
| :--- | :--- | :--- |
| 自编码器（AE/VAE） | 编码器压缩→解码器重建 | VAE、β-VAE |
| 掩码自编码器（MAE） | 随机遮挡，预测被遮挡部分 | MAE、BeiT |
| 隐式神经场（NeRF） | 用神经网络隐式表示3D场景 | NeRF、Instant-NGP |
| 4D重建 | 动态NeRF、4DGS | 4D Gaussian Splatting |

**工程注意点**：
- 理解层往往是整个系统中**显存消耗最大**的环节。
- 训练时，**掩码策略的设计**比模型架构选择更重要。
- 硬件瓶颈：显存容量 + 算力。

**典型代码骨架**（以MAE为例）：
```python
from models_mae import MaskedAutoencoderViT

model = MaskedAutoencoderViT(
    img_size=224, patch_size=16, in_chans=3,
    embed_dim=1024, depth=24, num_heads=16,
    decoder_embed_dim=512, decoder_depth=8, decoder_num_heads=16
)

# 输入图像 → 随机掩码75% → 重建完整图像
loss, pred, mask = model(image, mask_ratio=0.75)

# 提取编码器输出作为图像的表征
representation = model.forward_encoder(image, mask)
```

### 2.3 预测（Prediction）——“接下来是什么？”

**定义**：基于历史状态序列，外推未来的状态演变。回答“如果什么都不做，接下来会怎样”。

**输入输出示例**：

| 输入 | 输出 | 任务类型 | 现实案例 |
| :--- | :--- | :--- | :--- |
| 文本前半句“今天天气真” | “好” | 语言建模（续写） | 输入法联想、对话系统 |
| 过去30天股价 | 明日股价预测 | 时序预测 | 量化交易 |
| 视频前5帧 | 后3帧 | 视频预测 | 自动驾驶预判 |
| 代码前几行 | 后续代码 | 代码补全 | GitHub Copilot |

**工程本质**：学习时序依赖（Time Dependency）。根据过去推未来，假设演变规律是连续的、可外推的。模型本质上是在学习 `P(next | history)`。

**关键技术路线**：

| 路线 | 核心思想 | 代表方法 |
| :--- | :--- | :--- |
| 自回归（Autoregressive） | 逐个预测下一个Token | Transformer解码器（GPT） |
| 循环神经网络（RNN） | 通过循环连接保持隐藏状态 | LSTM、GRU |
| 状态空间模型（SSM） | 用状态方程描述序列演化 | Mamba、S4 |
| 潜空间预测 | 在压缩的潜空间中预测 | Sora（视频预测） |

**与推演的区分**：

| | 预测 | 推演 |
| :--- | :--- | :--- |
| 方法 | 统计外推 | 因果+物理计算 |
| 依据 | 历史数据中的统计模式 | 物理定律+逻辑规则 |
| 是否考虑干预 | 否（假设什么都不变） | 是（主动探索不同可能性） |
| 比喻 | “球在斜坡上，我猜它会滚下去” | “如果我在半路挡一块板子，它会弹向哪边？” |

**工程注意点**：
- 自回归推理是**串行的**，无法并行，这是GPU利用率低（30-40%）的根本原因。
- **投机采样（Speculative Decoding）** 是当前工程上缓解此问题的主流手段。
- 硬件瓶颈：显存带宽（HBM Bandwidth）。

**典型代码骨架**：
```python
from transformers import AutoModelForCausalLM, AutoTokenizer

model = AutoModelForCausalLM.from_pretrained("deepseek-ai/deepseek-7b")
tokenizer = AutoTokenizer.from_pretrained("deepseek-ai/deepseek-7b")

input_text = "今天天气真好，我"
inputs = tokenizer(input_text, return_tensors="pt")

# 自回归生成：循环预测下一个token
generated = model.generate(
    **inputs,
    max_new_tokens=100,  # 重复100次"预测下一个"
    temperature=0.7,
    do_sample=True
)

output_text = tokenizer.decode(generated[0])
# "今天天气真好，我打算出去散步..."
```

### 2.4 推演（Reasoning / Simulation）——“如果……会怎样？”

**定义**：主动构建虚拟场景，在内部推演不同选择的结果，选出最优方案。这是智能体从“被动反应”走向“主动决策”的分水岭。

**输入输出示例**：

| 输入 | 输出 | 任务类型 | 现实案例 |
| :--- | :--- | :--- | :--- |
| 围棋盘面+候选落子 | 最佳落子 | 棋类决策 | AlphaGo |
| 自动驾驶场景+候选路径 | 最优路径 | 路径规划 | 自动驾驶决策 |
| 用户问题+可用工具列表 | 该调用哪个工具 | Agent决策 | AI助手调用工具 |
| 4D场景+修改请求 | 重模拟后的结果 | 因果回溯 | 数字孪生推演 |

**工程本质**：状态空间搜索 + 因果推理 + 物理约束。与预测不同，推演可能依赖**确定性物理定律**（F=ma、动量守恒）或**逻辑规则**。

**关键技术路线**：

| 路线 | 核心思想 | 代表方法 |
| :--- | :--- | :--- |
| 树搜索 | 展开决策树，用评估函数剪枝 | MCTS（蒙特卡洛树搜索） |
| 物理引擎 | 用确定性物理定律模拟 | NVIDIA PhysX、Bullet |
| 因果图模型 | 用DAG表示因果关系 | Pearl因果推理 |
| 思维链 | 用LLM生成中间推理步骤 | OpenAI o1、CoT |

**工程注意点**：
- 推演层的核心往往是 **“模型+环境模拟器”的组合**。
- **搜索深度和宽度**直接决定决策质量和计算成本。
- 硬件瓶颈：算力 + 内存带宽。

**典型代码骨架**（MCTS核心循环）：
```python
class MCTSNode:
    def __init__(self, state, parent=None):
        self.state = state
        self.parent = parent
        self.children = []
        self.visits = 0
        self.value = 0.0

def mcts_search(root_state, num_simulations=1000):
    root = MCTSNode(root_state)
    for _ in range(num_simulations):
        node = select(root)           # 1. 选择
        if not is_terminal(node.state):
            node = expand(node)       # 2. 扩展
        reward = simulate(node.state) # 3. 模拟
        backpropagate(node, reward)   # 4. 回溯
    return best_child(root)

# 内部模拟了1000次"如果走A会怎样"、"如果走B会怎样"
# 然后选择最优的——这就是推演的本质
```

### 2.5 执行（Execution）——“我去改变它”

**定义**：将决策结果映射为对系统状态的实际变更。这是唯一一个对系统产生实际改变的环节。

**输入输出示例**：

| 输入 | 输出 | 任务类型 | 现实案例 |
| :--- | :--- | :--- | :--- |
| “查询北京的天气”决策 | 调用天气API | 数字执行 | 智能音箱查天气 |
| 最优动作向量 | 关节扭矩/角度指令 | 物理执行 | 工业机器人取物 |
| “修改文件属性”决策 | 写入.4dworld文件 | 数字执行 | 4D场景编辑 |

**工程本质**：将虚拟决策映射为物理或数字现实。

**关键技术路线**：

| 路线 | 代表工具 |
| :--- | :--- |
| 函数调用（Function Calling） | OpenAI Function Calling |
| 机器人操作系统 | ROS/ROS2、MoveIt |
| 物理引擎接口 | PhysX API、Bullet API |
| 实时控制 | FreeRTOS、VxWorks |

**执行层的关键特征**：
- 它是五个原子能力中**唯一对系统产生熵增或物理改变**的环节。
- 在数字世界，执行是“调用API/写入数据库”，毫秒级延迟可接受。
- 在物理世界（具身智能），执行是“驱动电机”，必须达到KHz级频率。
- 执行的**可靠性比智能更重要**：99%的正确率不够，物理世界无法“回滚”。

**工程注意点**：
- 执行层必须与推演层解耦——推演可以慢（Hz级），执行必须快（KHz级）。
- 硬件瓶颈：延迟 + 吞吐量。

**典型代码骨架**（ROS机器人执行）：
```python
import rospy
from geometry_msgs.msg import Twist

rospy.init_node('robot_executor')
cmd_pub = rospy.Publisher('/cmd_vel', Twist, queue_size=1)

def execute_action(action):
    cmd = Twist()
    cmd.linear.x = 0.5
    rate = rospy.Rate(1000)  # 1KHz
    for _ in range(1000):
        cmd_pub.publish(cmd)
        rate.sleep()
        if is_target_reached(get_feedback()):
            break
```

| 原子能力 | 根本问题 | 输入 | 输出 | 技术路线 | 代表模型 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 感知 | 现在是什么？ | 完整数据 | 标签/坐标 | CNN/ViT/PointNet | YOLO、ResNet |
| 理解 | 中间缺了什么？ | 不完整数据 | 完整结构 | AE/MAE/NeRF | BERT、NeRF、4DGS |
| 预测 | 接下来是什么？ | 历史序列 | 未来序列 | 自回归/RNN | GPT、LSTM、Sora |
| 推演 | 如果……会怎样？ | 状态+候选动作 | 最优决策 | 树搜索/物理引擎 | AlphaGo、o1、PhysX |
| 执行 | 我去改变它 | 决策结果 | 状态变更 | API调用/ROS | Agent、ROS |


## 3. 第二坐标轴：九大数据模态——它在处理什么数据原料

**决定AI代码第一行怎么写的，永远是数据的本质结构，而不是模型的名字。**

本报告按数据的“维度”和“结构复杂度”由低到高，将AI处理的所有数据归纳为9类。

### 3.1 第1类：纯文本（一维离散序列）

| 属性 | 描述 |
| :--- | :--- |
| 后缀 | .txt, .json, .py, .log, .md |
| 数据本质 | 按语法规则排列的离散符号串 |
| 计算机眼中的它 | 一串离散的ID（字典索引），如“你好”→[1024,2048] |

| 代表模型 | GPT系列、DeepSeek、Llama、BERT |
| 主要工程痛点 | 上下文窗口限制、幻觉问题 |
| 硬件瓶颈 | 显存带宽（搬运Token耗带宽，GPU算力常闲置） |
| 涉及原子能力 | 感知（情感分析）、理解（掩码建模）、预测（续写）、推演（思维链）、执行（工具调用） |

### 3.2 第2类：图像（二维密集矩阵）

### 3.3 第3类：音频（一维时间信号）

### 4.2 原子能力 × 评估指标速查表

| 原子能力 | 核心指标 | 工程含义 | 常见陷阱 |
| :--- | :--- | :--- | :--- |
| 感知 | Accuracy / mAP / F1 | 判断得准不准 | 用准确率评估不平衡数据 |
| 理解 | PSNR / SSIM / LPIPS | 重建得像不像 | 只看PSNR忽略人眼感知 |
| 预测 | Perplexity / RMSE | 猜得顺不顺 | 用分类指标评估回归任务 |
| 推演 | 胜率/成功率/物理偏差 | 选得优不优 | 忽略计算成本谈决策质量 |
| 执行 | 成功率 / P99延迟 / MTBF | 做得成不成、稳不稳 | 只测平均延迟不测P99 |

> **工程铁律**：不要用一个指标评估所有原子能力。感知看精度，预测看困惑度，推演看决策质量，执行看成功率。


## 5. 钱花在哪：硬件选型指南

### 5.1 三座大山

理解AI硬件的三个独立维度，是控制云账单的第一步：

| 维度 | 典型场景 | 核心硬件指标 | 选型策略 |
| :--- | :--- | :--- | :--- |
| **算力饥渴** | 图像识别/生成、3D重建 | GPU TFLOPS | 单卡FLOPS越高，吞吐越大 |
| **带宽饥渴** | LLM文本续写 | GPU HBM带宽（GB/s） | **优先看带宽，而非算力** |
| **显存容量饥渴** | 视频生成、4D重建 | GPU VRAM（GB） | 容量是硬门槛，不够就跑不动 |

**带宽饥渴型任务的特殊说明**：LLM推理时，GPU利用率常仅30-40%，瓶颈在显存搬运而非计算。A100（2TB/s带宽）vs H100（3.35TB/s带宽）——H100的67%带宽提升直接转化为吞吐量提升。

### 5.2 任务到硬件的映射表（2026年现状）

| 任务类型 | 数据模态 | 瓶颈 | 推荐配置 | 月成本（估算） | 备注 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 文本生成（7B） | 文本 | 带宽 | RTX 4090 24GB | ~$300-800 | 消费级卡可跑 |
| 文本生成（70B） | 文本 | 带宽+显存 | A100 80GB × 2卡 | ~$1,500-2,500 | 必须企业级卡 |
| 图像识别 | 图像 | 算力 | RTX 3060 12GB | ~$50-100 | 消费级足够 |
| 图像生成 | 图像 | 显存容量 | RTX 4060 Ti 16GB | ~$150-300 | 16GB是入门门槛 |
| 语音ASR/TTS | 音频 | 延迟 | CPU即可 | ~$10-50 | 不需要GPU |
| 视频生成 | 视频 | 显存容量 | A100 80GB × 4卡 | ~$3,000-6,000 | 只能云端 |
| 3D重建（NeRF） | 3D | 显存+算力 | RTX 4090 24GB | ~$200-400 | 24GB勉强跑 |
| 表格数据 | 表格 | CPU算力 | 4核8G云服务器 | ~$20-50 | **绝对不要用GPU** |
| 图网络（大图） | 图网络 | 分布式通信 | 多机多卡集群 | ~$1,000-3,000 | 消费级无法处理 |
| 多模态（VLA） | 混合 | 显存+带宽 | A100 80GB × 2卡 | ~$1,500-2,500 | 需加载多个编码器 |
| 4D世界训练 | 4D世界 | 三杀 | H100 80GB × 8卡+NVMe | ~$10,000-20,000 | 仅限实验室/大厂 |

**成本优化建议**：
- 文本生成（高吞吐）：使用vLLM或TensorRT-LLM，吞吐量提升2-3倍
- 图像生成（批量）：使用batch推理，单卡吞吐提升4-8倍
- 视频处理：用更低帧率（15fps）和更低分辨率
- 3D重建：使用Instant-NGP等加速方法
- 表格数据：**用CPU实例**（比GPU便宜5-10倍）


## 6. 接到需求怎么办：工程决策流程

接到AI项目后，按以下五步走，可绕过90%的无效讨论。

### 第一步：确定数据模态

问业务方：“原始文件的后缀名是什么？”

| 判断条件 | 归类 | 后续策略 |
| :--- | :--- | :--- |
| .txt/.json/.py/.md | 文本类 | 走LLM路线 |
| .jpg/.png | 图像类 | 走CV路线 |
| .wav/.mp3 | 音频类 | 走语音路线 |
| .mp4/.avi | 视频类 | 走视频路线 |
| .obj/.fbx/.ply | 3D模型类 | 走3D路线 |
| .csv/.xlsx | 表格类 | **直接锁定XGBoost** |
| 图数据库 | 图网络类 | 走GNN路线 |
| 混合输入 | 多模态类 | 走多模态路线 |
| 动态3D+时间轴 | 4D世界类 | 走4D路线（前沿） |

### 第二步：确定原子能力

问自己：“我要对这堆数据做什么？”

| 业务目标 | 原子能力 | 技术方向 |
| :--- | :--- | :--- |
| 判断“是什么/在哪里” | 感知 | 分类/检测/识别 |
| 从残缺数据恢复完整结构 | 理解 | 自编码器/NeRF |
| “接着往下写”或生成 | 预测 | 自回归/扩散 |
| “如果做X会怎样”做决策 | 推演 | 树搜索/物理引擎 |
| 实际去改变/操作 | 执行 | API调用/机器人控制 |

### 第三步：确定硬件瓶颈

根据第5章速查表确定瓶颈维度，锁定硬件配置。

### 第四步：确定自建 vs 调API

| 场景 | 建议 | 决策逻辑 |
| :--- | :--- | :--- |
| 文本生成，日调用 < 10万次 | 调API | 自建硬件成本 > API调用费 |
| 文本生成，日调用 > 10万次 | 自建（vLLM） | API调用费 > 硬件折旧成本 |
| 图像生成 | 优先调API | 自建需管理GPU集群 |
| 视频生成 | 只能调API | 自建集群百万级起步 |
| 表格数据 | 自建（XGBoost） | 成本极低，API反而贵 |
| 4D世界 | 实验室合作 | 尚无商业方案 |

### 第五步：确定评估指标

- **感知**：准确率、mAP、F1
- **理解**：PSNR、SSIM、重构误差
- **预测**：困惑度、RMSE
- **推演**：胜率、成功率、物理偏差度
- **执行**：成功率、P99延迟、MTBF


## 7. 新旧知识迁移：历史概念的重新对位

### 7.1 分类器 → 感知层的“离散输出”子类

| 传统概念 | 新框架位置 | 说明 |
| :--- | :--- | :--- |
| 分类器（SVM/决策树） | **感知** → 离散标签输出 | 输出是“猫/狗”概率分布 |
| 目标检测器 | **感知** → 坐标框+标签输出 | 输出是“物体在(x,y)” |
| 回归器（股价预测） | **感知** → 连续数值输出 | 一次推理出一个数 |

### 7.2 识别器 → 感知层的“特征比对”子类

| 传统概念 | 新框架位置 | 说明 |
| :--- | :--- | :--- |
| 人脸识别 | **感知** → 特征提取+比对 | 含特征向量提取+数据库检索 |
| 语音唤醒 | **感知** → 关键词匹配 | 识别特定语音指令 |

### 7.3 预测机 → 分为两类

| 传统概念 | 新框架位置 | 区分逻辑 |
| :--- | :--- | :--- |
| 数值预测（股价/销量） | **感知** → 回归 | 一次推理出一个数值 |
| 序列预测（LLM/自回归） | **预测** | 循环推理出一段序列 |

**区分口诀**：
- 一次推理出一个数 → 感知（回归）
- 循环推理出一段话 → 预测（自回归）

### 7.4 迁移路径总览

| 你过去的经验 | 在新框架中 | 你需要补充的知识 |
| :--- | :--- | :--- |
| 分类器（SVM） | 感知 → 分类 | 扩展到检测、分割、回归等 |
| 识别器（人脸） | 感知 → 识别 | 理解识别与理解的边界 |
| 数值预测（股价） | 感知 → 回归 | 区分“感知回归”与“预测自回归” |
| 时序预测（LSTM） | 预测 | 扩展到自回归语言模型 |
| 机器人控制（PID） | 执行 | 增加推演层（物理引擎）做规划 |


## 8. 前沿领域工程拆解：具身智能与4D世界

### 8.1 具身智能的工程实现路径

#### 8.1.1 一句话定义

> 具身智能 =（感知 + 理解 + 推演 + 执行）的实时物理闭环，其中“执行”从“可选项”升级为“刚性必选项”，响应频率从Hz级跃升至KHz级。

**拆解到两条线**：
- **数据**：4D世界（实时传感器流）→ 第9类数据模态
- **能力**：感知+理解+推演+执行 → 四个原子能力的实时协同

#### 8.1.2 数据模态：处理的是“实时4D世界流”

具身智能处理的数据属于第9类（4D世界）的**在线版本**——它不是静态文件，而是**连续实时传感器流**：

| 数据来源 | 具体内容 | 采样频率 |
| :--- | :--- | :--- |
| 视觉传感器 | RGB摄像头、深度摄像头、激光雷达 | 30-120Hz |
| 本体传感器 | 关节编码器、IMU惯性单元 | 500-1000Hz |
| 环境传感器 | 触觉、力觉、温度 | 100-1000Hz |

**工程铁律**：具身智能没有“文件后缀名”，输入是无穷无尽的带时间戳的传感器流。

#### 8.1.3 原子能力分层架构

| 层级 | 原子能力组合 | 具体任务 | 时间尺度 | 技术选型 |
| :--- | :--- | :--- | :--- | :--- |
| **大脑（决策层）** | 感知 → 理解 → 推演 | 识别障碍物→构建3D地图→规划路径 | Hz级（1-10Hz） | LLM/VLA模型 |
| **小脑（控制层）** | **执行（高频闭环）** | 生成电机指令、实时反馈调节 | KHz级（500-1000Hz） | PID/MPC/ROS |

> **关键设计原则**：大脑和小脑必须**完全解耦**。大脑可以慢思考（几百毫秒），小脑必须快速响应（毫秒级甚至微秒级）。这就是为什么特斯拉等公司采用“端到端神经网络（大脑）+ 传统控制算法（小脑）”的混合架构。

#### 8.1.4 具身智能 vs 纯数字AI：三个根本矛盾

| 矛盾 | 纯数字AI（如GPT） | 具身智能 | 工程后果 |
| :--- | :--- | :--- | :--- |
| **实时性矛盾** | 几百毫秒延迟可接受 | 每1毫秒需更新电机指令 | **必须分层解耦** |
| **物理真实性矛盾** | 数字世界是确定的 | 打滑、延迟、噪声、间隙 | 必须**在线学习** |
| **经验获取矛盾** | 从文本中学习 | 必须亲手“试”一百万次 | 需要**仿真训练** |

#### 8.1.5 借助LLM实现“理解-执行”闭环的工程架构

**完整链路**：
```
人类指令 → 感知（听懂）→ 理解（解析意图）→ 推演（任务拆解+规划）→ 执行（高频动作）→ 反馈学习
```

| 步骤 | 原子能力 | 技术实现 | 成熟度 |
| :--- | :--- | :--- | :--- |
| 1. 听懂指令 | 感知 | LLM语音/文本理解 | ✅ 成熟 |
| 2. 推断隐藏目标 | 理解 | 社会推理+上下文建模 | ⚠️ 较成熟 |
| 3. 分解长程任务 | 推演 | 思维链/规划链 | ✅ 较成熟 |
| 4. 感知环境状态 | 理解（视觉） | VLA视觉编码器 | ⚠️ 较成熟 |
| 5. 匹配规划与场景 | 推演（验证） | 场景-任务匹配验证 | ⚠️ 较成熟 |
| 6. 生成控制指令 | 执行（控制层） | ROS-LLM/动作思维链 | ❌ **核心瓶颈** |
| 7. 监控执行状态 | 感知（反馈） | 关节编码器/IMU读取 | ✅ 成熟 |
| 8. 修正执行策略 | 推演（在线学习） | 失败归因+策略调整 | ❌ 前沿 |

**核心技术路径**：**VLA（Vision-Language-Action）模型**——将大语言模型的泛化能力应用于物理世界行为决策。

#### 8.1.6 当前工程成熟度（2026年）

| 环节 | 成熟度 | 说明 |
| :--- | :--- | :--- |
| 意图理解 | ✅ 成熟 | LLM基础能力已足够 |
| 任务拆解 | ✅ 较成熟 | 思维链可支撑长任务 |
| 感知对齐 | ⚠️ 较成熟 | VLA已落地，仍需精调 |
| 动作执行 | ❌ **核心瓶颈** | 语义-运动鸿沟待攻克 |
| 在线学习 | ❌ 前沿 | 需更多真实数据积累 |

**真实落地案例**：华为ROS-LLM框架已能驱动机器人**自主完成“煮咖啡”（12步骤）**——五原子能力全链路的完整实例。


### 8.2 4D世界的工程架构

#### 8.2.1 场景定义

用户上传一段环绕拍摄的视频（如球在桌面上滚动）→ 系统生成 `.4dworld` 文件 → 支持：
1. 任意时间点自由旋转视角观看
2. 拖拽时间轴回到任意时刻
3. 修改历史中某物体的属性（如木箱→铁箱）
4. 系统自动重新模拟后续演变

#### 8.2.2 技术拆解（按原子能力分层）

| 步骤 | 原子能力 | 具体技术 | 成熟度 |
| :--- | :--- | :--- | :--- |
| 1. 视频→动态3D场景 | **理解** | 4D Gaussian Splatting | 前沿 |
| 2. 识别场景物体 | **感知** | 3D-BEV / PointNet++ | 较成熟 |
| 3. 估计物理属性 | **理解** | 物理参数反演 | 前沿（实验室） |
| 4. 预测未来运动 | **预测** | 物理外推模型 | 较成熟 |
| 5. 用户拖回第3秒 | **推演** | 状态回退+因果日志 | 可设计 |
| 6. 修改“木箱→铁箱” | **执行** | 属性写入+触发重模拟 | 可设计 |
| 7. 重算后续演变 | **推演** | 物理引擎重跑（PhysX） | ✅ **成熟** |
| 8. 渲染播放 | **预测**（渲染） | 高斯泼溅实时渲染 | 较成熟 |

#### 8.2.3 工程架构（逻辑层）

```
用户交互层（时间轴拖拽 / 属性修改 / 视角控制）
            ↓
执行层（属性写入 / 触发重模拟 / 因果日志）
            ↓
推演层（物理引擎重模拟 / 因果回溯）
            ↓
预测层（物理外推 / 视觉渲染）
            ↓
理解层（4D场景表征 / 物理参数反演）
            ↓
感知层（物体识别 / 状态估计）
            ↓
数据层（.4dworld：几何 + 状态序列 + 因果日志）
```

#### 8.2.4 能力成熟度与可用工具

| 所需能力 | 原子能力 | 成熟度 | 可用工具 |
| :--- | :--- | :--- | :--- |
| 4D重建 | 理解 | 前沿 | 4DGS、动态NeRF |
| 场景感知 | 感知 | 较成熟 | 3D-BEV、PointNet++ |
| 物理属性反演 | 理解 | 前沿 | 实验室自研 |
| 物理外推 | 预测 | 较成熟 | LSTM、外推模型 |
| 因果日志回溯 | 推演 | 可设计 | 类Git版本管理 |
| 重模拟 | 执行→推演 | **成熟** | PhysX、Bullet |
| 实时渲染 | 预测（渲染） | 较成熟 | 3DGS实时渲染 |

## 10. 结语与实践建议

### 10.1 核心框架回顾

> **AI = 5个原子能力 × 9种数据模态**
>
> - **5个原子能力**：感知、理解、预测、推演、执行
> - **9种数据模态**：文本、图像、音频、视频、3D模型、表格、图网络、多模态、4D世界
>
> 任意AI系统，都可以被唯一映射到这个 **5×9=45个格子** 的坐标系中。

### 10.2 应用方法：两层过滤法

面对任何AI论文、产品或项目，执行两层过滤：

**第一层（数据模态）**：“它处理的数据是九类中的哪一类？”
→ 锁定输入层技术栈和预处理方式

**第二层（原子能力）**：“它调用的是五个原子能力中的哪几个？”
→ 锁定核心算法类型和硬件瓶颈

两层过滤完成，该系统的底层技术本质对你完全透明。剩下的只是工程优化和成本核算。
