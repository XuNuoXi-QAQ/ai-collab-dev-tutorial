## 6.1 团队协作概述

### 6.1.1 为什么需要协作规范？

**没有规范的团队会遇到的问题：**

- 🎭 **代码风格混乱**：每个人写的代码都不一样
- 🐛 **Bug 频繁**：质量没有保障
- 📉 **效率低下**：沟通成本高，重复劳动多
- 😤 **冲突不断**：代码合并冲突，意见不合
- 📚 **知识孤岛**：只有某些人懂某些模块

**好的协作规范带来的价值：**

| 价值 | 说明 |
|------|------|
| ⚡ **提升效率** | 减少沟通成本，加快开发速度 |
| 🎯 **保证质量** | 统一标准，代码质量更稳定 |
| 🤝 **减少冲突** | 明确流程，减少摩擦 |
| 📚 **知识共享** | 团队共同成长，避免单点依赖 |
| 😊 **提升士气** | 工作更顺畅，团队更和谐 |

### 6.1.2 高效团队的特征

- ✅ **清晰的目标**：每个人都知道要做什么
- ✅ **明确的分工**：谁负责什么很清楚
- ✅ **顺畅的沟通**：信息流通，及时同步
- ✅ **统一的标准**：代码、流程、文档都有规范
- ✅ **互相尊重**：尊重彼此的意见和工作
- ✅ **持续改进**：定期回顾，不断优化

## 6.2 代码规范

### 6.2.1 为什么需要代码规范？

- 👀 **可读性**：所有人的代码风格一致，容易读懂
- 🔧 **可维护性**：别人接手代码更容易
- 🐛 **减少 Bug**：规范的代码出问题的概率更低
- ⚡ **提升效率**：不用纠结风格问题，专注业务逻辑

### 6.2.2 代码规范的内容

**1. 命名规范**

```
# 变量和函数：小驼峰（JavaScript）或蛇形（Python）
userName / user_name

# 类名：大驼峰
UserProfile

# 常量：全大写 + 下划线
MAX_RETRY_COUNT

# 布尔值：用 is/has/can 开头
isActive / hasPermission / canEdit
```

**2. 代码格式**

- 缩进：2 空格或 4 空格，统一即可
- 行宽：建议 80-120 字符
- 空行：逻辑块之间加空行
- 括号：统一风格（同列或尾随行末）

**3. 注释规范**

- 注释为什么（why），而不是做什么（what）
- 公共 API 必须有文档注释
- 复杂逻辑必须加注释
- 及时更新注释，避免过时

**4. 文件组织**

- 按功能模块组织目录
- 一个文件一个主要类/功能
- 文件名清晰表达内容

### 6.2.3 自动化工具

**不要靠人来检查规范，要用工具：**

| 语言 | 代码风格 | 静态检查 | 格式化 |
|------|----------|----------|--------|
| JavaScript | ESLint | ESLint | Prettier |
| TypeScript | ESLint | ESLint | Prettier |
| Python | PEP 8 | flake8 / pylint | Black |
| Go | Go 官方规范 | go vet | gofmt |
| Java | - | Checkstyle | - |

**最佳实践：**
- ✅ 提交前自动格式化
- ✅ CI 中自动检查
- ✅ 不通过检查不让合并

## 6.3 Git 工作流规范

### 6.3.1 分支规范

**分支命名：**

```
# 功能分支
feature/功能描述
feature/user-login
feature/payment-integration

# Bug 修复
fix/问题描述
fix/login-error
fix/issue-123

# 热修复
hotfix/版本号-描述
hotfix/v1.2.1-security

# 发布分支
release/版本号
release/v1.0.0

# 文档
docs/描述
docs/api-guide
```

**分支保护：**

- main/master 分支必须保护
- 不允许直接 push 到主分支
- 必须通过 PR 合并
- 必须通过 CI 检查
- 必须至少 1 人审查通过

### 6.3.2 提交规范

**Commit Message 规范（Conventional Commits）：**

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type 类型：**

| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | Bug 修复 |
| docs | 文档更新 |
| style | 代码格式调整 |
| refactor | 代码重构 |
| perf | 性能优化 |
| test | 测试相关 |
| chore | 构建/工具等杂项 |

**示例：**

```
feat(auth): 添加用户登录功能

- 实现用户名密码登录
- 添加登录失败次数限制
- 集成 JWT token 认证

Closes #123
```

**好的提交信息的特点：**
- ✅ 清晰说明做了什么
- ✅ 一个提交只做一件事
- ✅ 用动词开头（添加、修复、优化...）
- ✅ 关联相关 Issue

### 6.3.3 PR 规范

**PR 大小：**
- 理想：200-400 行代码
- 最大：不超过 800 行
- 太大的 PR 要拆分

**PR 描述模板：**
```markdown
## 变更说明

## 变更类型
- [ ] feat: 新功能
- [ ] fix: Bug 修复
- [ ] docs: 文档
- [ ] refactor: 重构
- [ ] test: 测试
- [ ] chore: 杂项

## 关联 Issue
Closes #

## 测试情况
- [ ] 单元测试
- [ ] 集成测试
- [ ] 手动测试

## 截图（如适用）
```

**审查要求：**
- 至少 1 个审查者批准
- 所有意见都已处理
- CI 全部通过
- 没有合并冲突

## 6.4 Issue 管理规范

### 6.4.1 Issue 类型

| 类型 | 标签 | 说明 |
|------|------|------|
| Bug | bug | 功能不正常 |
| 新功能 | feature | 新的需求 |
| 优化 | enhancement | 现有功能改进 |
| 文档 | documentation | 文档相关 |
| 疑问 | question | 问题咨询 |
| 任务 | task | 具体的开发任务 |

### 6.4.2 Issue 模板

**Bug 报告模板：**
```markdown
## 问题描述

## 复现步骤
1. 
2. 
3. 

## 预期行为

## 实际行为

## 环境信息
- 系统：
- 浏览器：
- 版本：

## 截图/日志
```

**功能需求模板：**
```markdown
## 功能描述

## 背景/动机

## 期望的行为

## 验收标准
- [ ] 
- [ ] 
- [ ] 

## 相关资源
```

### 6.4.3 Issue 生命周期

```
新建 (New)
   ↓
确认有效 (Confirmed)
   ↓
分配负责人 (Assigned)
   ↓
进行中 (In Progress)
   ↓
待验证 (Ready for Test)
   ↓
已完成 (Closed)
```

## 6.5 代码审查规范

### 6.5.1 审查者的职责

- ✅ 及时审查：24 小时内响应
- ✅ 认真负责：仔细看，不走过场
- ✅ 给出建议：不仅说不好，还要说怎么改
- ✅ 尊重作者：对事不对人，礼貌沟通

### 6.5.2 被审查者的职责

- ✅ 自测代码：提交前自己先测
- ✅ 写好描述：让审查者快速理解
- ✅ 及时响应：审查反馈尽快处理
- ✅ 虚心接受：把审查当学习机会

### 6.5.3 审查清单

**功能正确性：**
- [ ] 逻辑是否正确？
- [ ] 边界条件是否考虑？
- [ ] 错误处理是否完善？

**代码质量：**
- [ ] 命名是否清晰？
- [ ] 有没有重复代码？
- [ ] 函数是否足够小？
- [ ] 注释是否恰当？

**安全性：**
- [ ] 输入验证？
- [ ] 权限控制？
- [ ] 敏感信息保护？

**性能：**
- [ ] 有没有明显的性能问题？
- [ ] 数据库查询是否优化？

**测试：**
- [ ] 有没有单元测试？
- [ ] 覆盖主要场景？

## 6.6 文档规范

### 6.6.1 需要哪些文档？

| 文档 | 说明 | 谁来写 |
|------|------|--------|
| README | 项目介绍、快速开始 | 项目负责人 |
| API 文档 | 接口说明 | 后端开发 |
| 架构文档 | 系统设计、技术选型 | 架构师 |
| 开发指南 | 环境搭建、开发流程 | 技术负责人 |
| 部署文档 | 部署步骤、运维说明 | 运维/DevOps |
| 变更日志 | 版本更新记录 | 每次发版更新 |

### 6.6.2 文档写作原则

- ✅ **面向读者**：考虑读者是谁，他们需要什么
- ✅ **清晰简洁**：用简单的话，不要太技术化
- ✅ **结构清晰**：有目录，有层级
- ✅ **及时更新**：代码变了，文档也要更
- ✅ **示例丰富**：用例子说明更容易懂

### 6.6.3 README 的标准结构

```
# 项目名称

## 简介
一句话说明这个项目是做什么的

## 功能特性
- 特性 1
- 特性 2

## 快速开始
### 环境要求
### 安装步骤
### 使用示例

## 文档
链接到详细文档

## 贡献指南
如何参与贡献

## 许可证
MIT License
```

## 6.7 团队沟通规范

### 6.7.1 沟通渠道

| 场景 | 推荐渠道 | 原因 |
|------|----------|------|
| 代码讨论 | GitHub PR/Issue | 可追溯，与代码关联 |
| 日常沟通 | 即时通讯（飞书/钉钉） | 快速响应 |
| 正式决策 | 邮件/会议纪要 | 正式记录 |
| 知识分享 | Wiki/文档 | 沉淀积累 |
| 紧急问题 | 电话/语音 | 最快解决 |

### 6.7.2 沟通原则

- ✅ **异步优先**：能用异步就不用同步
- ✅ **信息完整**：一次说清楚，不要挤牙膏
- ✅ **及时响应**：看到消息尽快回复
- ✅ **公开透明**：能公开就不私聊
- ✅ **尊重时间**：不要随便打断别人

### 6.7.3 会议规范

**开会前：**
- 有明确的议题
- 提前发会议邀请和材料
- 只邀请相关的人

**开会中：**
- 准时开始，准时结束
- 有人主持，控制节奏
- 有人记录，形成结论

**开会后：**
- 24 小时内发出会议纪要
- 明确行动项、负责人、截止时间
- 跟进执行情况

## 6.8 版本发布规范

### 6.8.1 语义化版本（Semantic Versioning）

```
主版本号.次版本号.修订号
MAJOR.MINOR.PATCH
```

- **主版本号**：不兼容的 API 变更
- **次版本号**：向下兼容的功能性新增
- **修订号**：向下兼容的问题修正

**示例：**
- `1.0.0` → `1.0.1`：修复 Bug
- `1.0.0` → `1.1.0`：新增功能
- `1.0.0` → `2.0.0`：不兼容的大改

### 6.8.2 发布流程

```
1. 确定发布内容
   ↓
2. 创建发布分支
   ↓
3. 冻结功能，只修 Bug
   ↓
4. 测试验证
   ↓
5. 准备发布文档
   ↓
6. 正式发布
   ↓
7. 合并回主分支
   ↓
8. 监控和跟进
```

### 6.8.3 发布检查清单

- [ ] 所有功能都已完成并测试
- [ ] 所有 Bug 都已修复
- [ ] 文档已更新
- [ ] 变更日志已写
- [ ] 数据库迁移已准备
- [ ] 回滚方案已准备
- [ ] 相关人员已通知

## 6.9 建立团队规范的步骤

### 6.9.1 制定规范

1. **收集意见**：听听大家的想法
2. **参考最佳实践**：学习业界成熟的做法
3. **结合实际**：适合自己团队的才是最好的
4. **达成共识**：大家都同意才能执行
5. **写成文档**：白纸黑字，有据可依

### 6.9.2 推行规范

1. **培训宣导**：让每个人都知道
2. **工具辅助**：自动化检查，减少人工
3. **以身作则**：Leader 带头遵守
4. **耐心帮助**：新人要有适应期
5. **定期回顾**：规范也需要迭代优化

### 6.9.3 持续改进

- 定期回顾规范的执行情况
- 收集大家的反馈和建议
- 根据实际情况调整优化
- 好的规范是演进出来的

## 6.10 实战练习

### 练习：为你的团队制定一份协作规范

**思考以下问题：**

1. 你们团队目前在协作上遇到了哪些问题？
2. 哪些问题可以通过制定规范来解决？
3. 你们团队最需要哪些规范？（代码、Git、流程...）
4. 如何让大家愿意遵守这些规范？

**可以从简单的开始：**
- 先统一代码风格
- 再规范 Git 提交
- 然后完善 PR 流程
- 逐步建立完整的规范体系
